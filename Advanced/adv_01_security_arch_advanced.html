— Enterprise Security Architecture Frameworks
SABSA, TOGAF, and NIST CSF applied to security architecture design
SABSA: risk-driven architecture from business context to technical controls
The Sherwood Applied Business Security Architecture (SABSA) framework structures security architecture across six layers, from the business context at the top through conceptual, logical, physical, component, and operational layers. Each layer asks who, what, how, when, where, and why at its level of abstraction. The business layer establishes the risk context — what assets matter, what threats are relevant, what the organization's risk tolerance is. The conceptual layer defines security policies and principles derived from the business context. The logical layer designs security services and their relationships. The physical layer specifies technologies implementing the logical services. The component layer details specific products and configurations. The operational layer defines security management processes. SABSA prevents the common architectural failure of designing technical controls without tracing them to business objectives — every control has a traceable justification from business risk, which enables prioritization when resources are constrained and defensible documentation for regulatory examination.
NIST Cybersecurity Framework 2.0: governance, identify, protect, detect, respond, recover
NIST CSF 2.0 expanded the original five functions to six by adding Govern, which establishes the organizational context for cybersecurity risk management — policies, roles, responsibilities, and oversight mechanisms that create the conditions for effective cybersecurity across all other functions. Identify focuses on understanding the organization's assets, risks, and vulnerabilities. Protect implements controls to limit or contain the impact of cybersecurity events. Detect develops capabilities to identify cybersecurity events in a timely manner. Respond takes action regarding a detected cybersecurity incident. Recover maintains plans for resilience and restores capabilities impaired by incidents. Advanced security architecture uses the CSF as both a maturity assessment framework — evaluating current capabilities against desired target states — and a communication tool for executive and board-level reporting that translates technical security status into business risk language.
Security architecture maturity models: measuring and improving
Security architecture maturity models provide a framework for assessing where an organization's security architecture currently stands and where it should go. The CMMI-derived model uses five levels: Initial (ad hoc, inconsistent), Managed (basic processes exist but inconsistently applied), Defined (standardized processes documented and consistently followed), Quantitatively Managed (processes measured and controlled through metrics), and Optimizing (continuous improvement based on measurement). For security architecture specifically, maturity assessment evaluates: documentation completeness (are architecture designs documented at all layers), consistency of application (are architecture standards applied consistently or only to high-visibility systems), measurement (are security architecture outcomes measured), and continuous improvement processes (do post-incident reviews and threat intelligence update the architecture). Maturity assessment provides objective basis for investment decisions — organizations at Level 1 benefit more from foundational documentation than from advanced tooling.
Reference architectures: industry patterns for common security problems
Reference architectures provide validated architectural patterns for common security problems, reducing the need to design solutions from scratch and providing benchmarks against which existing architectures can be assessed. The NIST National Cybersecurity Center of Excellence (NCCoE) publishes reference architectures for specific domains including zero trust, identity and access management, and data classification. The Cloud Security Alliance provides reference architectures for cloud security across IaaS, PaaS, and SaaS models. CISA publishes reference architectures for critical infrastructure sectors. For insurance agencies, relevant reference architectures cover: remote access for a distributed workforce, secure integration with carrier systems and rating platforms, and SOC architecture for organizations with limited internal security operations capability. Using reference architectures accelerates design, provides defensible architectural decisions, and ensures alignment with industry practice that regulators recognize.
— Resilient Architecture Design
Designing systems that withstand sophisticated attacks
Blast radius minimization: designing for breach containment
Blast radius minimization is the architectural principle of designing every system component as if it will be compromised, and ensuring that its compromise provides the attacker with the minimum possible access to additional systems and data. The implementation requires answering — for every system component — the question: if this component is completely controlled by an attacker, what can they reach from here? The answer drives segmentation decisions: components that would provide broad access if compromised should be isolated behind additional authentication and access control boundaries. Privileged access workstations have minimal network connectivity because their compromise would provide access to the most sensitive systems. Secrets management systems are isolated from application servers, requiring explicit credential checkout rather than storing credentials locally. Backup systems are isolated from production systems using credentials that are not used anywhere in the production environment, because backup systems are a ransomware target specifically because their compromise eliminates recovery capability.
Immutable infrastructure and infrastructure as code security
Immutable infrastructure is the architectural pattern of treating deployed servers and containers as disposable units that are never modified after deployment — when a change is needed, a new instance is deployed from the current approved image and the old instance is terminated, rather than patching or modifying the running instance. Security benefits are significant: configuration drift (unauthorized or unintended changes to running systems) is eliminated because any deviation from the approved image is resolved at next deployment; incident response is simplified because recovering from compromise means replacing the instance from a known-good image rather than hunting for all persistence mechanisms; and the deployment pipeline becomes the security control boundary, where all changes must pass through code review, security scanning, and approved build processes. Infrastructure as code (IaC) tools — Terraform, CloudFormation, Pulumi — store infrastructure definitions in version-controlled repositories where security controls (policy as code, automated security scanning) can be applied before deployment.
Chaos engineering for security: testing resilience assumptions
Chaos engineering originated as a reliability practice — deliberately introducing failures into production systems to verify that resilience designs work as expected under real conditions rather than theoretical ones. Applied to security, chaos engineering tests security resilience assumptions: does network isolation actually prevent lateral movement, or are there overlooked connectivity paths? Does the incident response procedure actually work when a critical system is unavailable, or does the response depend on that system? Does the backup restoration procedure actually complete within the recovery time objective, or has configuration drift made the procedure slower than documented? Security chaos experiments might include: simulating a ransomware infection on a specific network segment to test whether isolation prevents spread, disabling the primary authentication provider to test whether fallback procedures maintain security, or testing whether the incident response playbook for a specific scenario produces the expected actions from the response team.
Supply chain security architecture: third-party component risk
Modern software and systems are built on layers of third-party components — open source libraries, commercial frameworks, vendor APIs, cloud platform services. Each layer introduces supply chain risk: a vulnerability or compromise in a dependency can affect every system that uses it, as demonstrated by the SolarWinds and Log4Shell incidents. Supply chain security architecture addresses this through: software composition analysis (automated scanning of application dependencies for known vulnerabilities), software bill of materials (SBOM) documentation of all components in deployed software, provenance verification (cryptographic verification that software artifacts originate from trusted sources and have not been modified), vendor security assessment (evaluating the security practices of critical software and service vendors), and isolation of third-party components (running third-party code in sandboxed environments with minimal access to sensitive systems and data). NIST SP 800-161r1 provides comprehensive guidance on supply chain risk management for federal systems, with applicability to regulated private sector organizations.
— Security Architecture for Cloud-Native Environments
Applying architecture principles to cloud and hybrid deployments
Cloud security posture management at scale
Cloud environments introduce architectural security challenges that do not exist in traditional on-premises architectures: configuration changes can be made through APIs by any authorized user without the change management friction of physical infrastructure, the scale of cloud deployments means thousands of resources may need configuration assessment, and the shared responsibility model means that the provider handles infrastructure security while the customer is responsible for everything built on that infrastructure. Cloud Security Posture Management (CSPM) platforms continuously assess cloud resource configurations against security policies and compliance frameworks, providing real-time visibility into misconfigurations across multi-cloud deployments. Advanced CSPM capabilities include: attack path analysis (identifying the sequence of misconfigurations an attacker could chain to reach sensitive resources), infrastructure drift detection (alerting when deployed resources diverge from their IaC definitions), and automated remediation (applying corrective configurations automatically for high-confidence, low-risk findings).
Container and Kubernetes security architecture
Container architectures introduce a distinct security model where the traditional boundaries — separate virtual machines with separate operating systems — are replaced by containers sharing a host kernel. A vulnerability in the container runtime or the host kernel can provide an attacker who has compromised any container with access to all containers on the same host, making container isolation security-critical. Kubernetes security architecture requires: pod security standards (restricting what capabilities and privileges containers can request), network policies (restricting east-west traffic between pods, applying the same micro-segmentation principles as traditional network segmentation), image security (scanning container images for vulnerabilities before deployment, signing images to verify provenance, using minimal base images that reduce attack surface), secrets management (never storing secrets in environment variables or container images, using Kubernetes secrets or external secrets managers with appropriate RBAC), and runtime security (detecting anomalous container behavior such as process execution outside the defined application pattern or unexpected network connections).
Serverless and function-as-a-service security architecture
Serverless architectures eliminate the server management burden but introduce architectural security patterns that differ significantly from traditional server-based deployments. Serverless functions execute in ephemeral environments that are terminated after execution, which eliminates some traditional security concerns (no long-lived server to compromise, no persistent malware) but introduces others. Each function is a separate authorization boundary requiring its own IAM policy — the principle of least privilege applied at the function level means each function has permissions only for the specific AWS services, databases, and APIs it needs. Over-permissioned serverless functions are the most common serverless security finding. Event injection attacks target the data inputs that trigger function execution — serverless functions process events from diverse sources (API calls, message queues, file uploads, database changes) and must validate every input source as untrusted. The ephemeral execution model means runtime monitoring must be done through log analysis and tracing rather than traditional endpoint agents.
Security architecture for hybrid environments: on-premises and cloud integration
Hybrid environments — where workloads span on-premises infrastructure and cloud platforms — create architectural complexity in identity, network, and data security. Identity federation extends the on-premises directory to cloud environments: users authenticate to the on-premises identity provider and receive tokens accepted by cloud services. The security of the federation trust determines the security of both environments — if the on-premises identity provider is compromised, all federated cloud resources are also accessible. Network connectivity between on-premises and cloud environments creates bidirectional attack paths: an attacker who compromises an on-premises system can reach cloud resources through the connectivity, and vice versa. Data governance must span both environments: data classification policies, encryption requirements, and access controls must be consistently applied whether data resides on-premises or in cloud storage. Security monitoring must consolidate telemetry from both environments into a unified view — siloed on-premises and cloud monitoring creates blind spots at the boundary.
— Security Architecture Governance
Sustaining and evolving security architecture in the organization
Architecture review boards: decision authority and process
An architecture review board (ARB) is the governance body responsible for evaluating proposed system designs, technology selections, and significant changes against the organization's architecture standards and principles. For security, the ARB (or a security architecture review process embedded within it) ensures that security requirements are addressed in design decisions before implementation, that new technologies meet security standards before adoption, and that deviations from architecture standards are explicitly approved with documented rationale and compensating controls. Effective security architecture review processes define: which changes require review (all new systems, integrations with external parties, changes to data flows involving NPI), what the review evaluates (alignment with security principles, threat model completeness, control adequacy for identified threats), who conducts the review (security architects or security team members with architecture expertise), and what the output is (approved with conditions, approved with compensating controls, or rejected with required remediation).
Security architecture metrics: measuring effectiveness
Security architecture effectiveness is difficult to measure directly — the absence of incidents does not prove that the architecture is sound, because an absence of detected incidents could reflect either good security or poor detection. Indirect metrics provide better signals. Control coverage metrics measure what fraction of systems meet architecture standards — 95% of servers with EDR installed, 88% of APIs with authentication enforced. Technical debt metrics measure the gap between current architecture and the target state — number of systems using deprecated TLS versions, number of firewall rule sets that have not been audited in the past year. Architecture alignment metrics measure whether new deployments meet standards at launch — percentage of new projects that completed architecture review, percentage of new cloud resources deployed through approved IaC templates. Post-incident architecture attribution identifies what architectural weakness the incident exploited, enabling measurement of whether architecture improvements actually close the gaps that incidents reveal.
Communicating security architecture to leadership and boards
Security architecture decisions have significant business implications — they determine what is possible, what the risk exposure is, and what it costs to achieve a given security posture. Communicating these decisions effectively to non-technical leadership requires translation from technical architecture language to business risk language. The board does not need to understand micro-segmentation; they need to understand that without it, a single compromised workstation provides access to the entire network, and that implementing it reduces the blast radius of any breach to the segment containing the compromised system. Architecture maturity levels translate to risk exposure statements: organizations at maturity Level 1 face documented risk categories that differ from Level 3 organizations. Investment proposals for architecture improvements should present the current risk exposure, the risk reduction achieved by the proposed investment, and the residual risk remaining after investment — a framework that enables leadership to make informed decisions about acceptable risk rather than technical decisions about security tools.
Continuous architecture improvement: feedback loops and learning
Security architecture improvement requires closing the feedback loop between what the architecture is designed to do and what it actually does in practice. Four feedback mechanisms drive continuous improvement. Post-incident architecture reviews identify which architectural assumptions were violated or which controls failed during the incident, producing architectural changes rather than just operational remediations. Threat intelligence updates feed new attacker techniques into the threat model, identifying where current controls are inadequate for current threats. Red team and penetration testing findings reveal which controls work as designed and which have exploitable weaknesses not apparent from architecture review alone. Security metrics trends identify controls that are consistently underperforming, driving investigation into whether the control design, implementation, or configuration is the root cause. The output of each feedback mechanism is an architecture backlog — documented architectural improvements prioritized by risk reduction — that drives the investment decisions for the next architecture evolution cycle.
— Summary
Key concepts from this module
SABSA traces controls to business risk; NIST CSF 2.0 adds Govern; maturity models enable investment prioritization
SABSA's six layers ensure every control has traceable business justification. NIST CSF 2.0's Govern function establishes organizational context for all other functions. Maturity assessment produces objective basis for investment decisions — foundational gaps warrant different investments than advanced capability gaps.
Blast radius minimization, immutable infrastructure, chaos engineering, and supply chain security address sophisticated threats
Every component should be designed assuming compromise — minimize what an attacker gains from any single system. Immutable infrastructure eliminates configuration drift and simplifies incident response. Chaos engineering tests resilience assumptions under real conditions. SBOM and provenance verification address the supply chain attack surface.
Cloud-native architecture requires CSPM, container security, serverless IAM, and hybrid identity federation integrity
CSPM provides continuous misconfiguration detection with attack path analysis. Container security requires pod standards, network policies, image signing, and runtime monitoring. Serverless functions require per-function least-privilege IAM. Hybrid environments require unified monitoring spanning both environments and protection of the federation trust as the highest-value target.
Architecture governance through ARB, metrics, leadership communication, and feedback loops sustains security posture
ARBs ensure security review before implementation. Control coverage and alignment metrics measure effectiveness. Board communication translates architecture to risk exposure language. Post-incident reviews, threat intelligence, and red team findings create the feedback loop that drives continuous architectural improvement.
