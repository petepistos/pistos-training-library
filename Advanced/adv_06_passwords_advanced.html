— Advanced Credential Attacks
Sophisticated techniques targeting enterprise authentication
DCSync and Golden Ticket attacks: Active Directory credential attacks
DCSync is a post-exploitation technique that abuses the Active Directory replication protocol to extract password hashes from a domain controller without requiring interactive access to the DC itself. An attacker with DS-Replication-Get-Changes and DS-Replication-Get-Changes-All permissions — which are legitimately held by domain controllers and by accounts assigned the Backup Operators or Domain Admins group — can use Mimikatz's DCSync module to request the password hash of any account in the domain, including the KRBTGT account. The KRBTGT password hash enables the Golden Ticket attack: using the KRBTGT hash to forge Kerberos Ticket Granting Tickets for any user in the domain with any group membership, with any validity period. A Golden Ticket can be forged for a non-existent user with Domain Admin membership, valid for ten years, that will be accepted as legitimate by any Kerberos service in the domain until the KRBTGT password is rotated twice.
NTLM relay and credential coercion techniques
NTLM relay attacks intercept NTLM authentication challenges from a victim system and relay them to a target service, using the victim's credentials to authenticate without ever cracking the hash. The attack requires coercing a target system to authenticate to an attacker-controlled server — techniques including PetitPotam (exploiting the EFS RPC interface), PrinterBug/SpoolSample (exploiting the Print Spooler service), and DFSCoerce (exploiting Distributed File System) trigger NTLM authentication from target systems including domain controllers. The relayed authentication can be used to: authenticate to LDAP (enabling creation of computer accounts, modification of ACLs, or Kerberos RBCD abuse), authenticate to SMB (enabling file access or code execution on the target), or authenticate to web services using NTLMAuth. NTLM relay mitigation requires: enabling LDAP signing and channel binding, enabling SMB signing on all servers, and disabling NTLM where Kerberos can be used exclusively.
OAuth and OIDC token abuse: modern authentication attack patterns
Modern applications increasingly use OAuth 2.0 and OpenID Connect (OIDC) for authentication and authorization rather than traditional password-based authentication. These protocols introduce distinct attack surfaces. Refresh token theft: OAuth refresh tokens are long-lived credentials that allow obtaining new access tokens without re-authenticating; an attacker who obtains a refresh token maintains persistent access until it expires or is revoked. Token smuggling: access tokens obtained through phishing can be replayed from attacker infrastructure, bypassing MFA because authentication has already occurred. OAuth authorization code interception: vulnerabilities in OAuth implementation (missing state parameter, PKCE bypass, open redirect exploitation) allow interception of authorization codes that can be exchanged for access tokens. For Microsoft 365, the Primary Refresh Token (PRT) stored in Windows Credential Manager provides SSO capability across all Azure AD-integrated applications — PRT theft provides comprehensive access equivalent to full session compromise.
Passwordless authentication risks and implementation pitfalls
Passwordless authentication — using FIDO2 passkeys, Windows Hello for Business, or certificate-based authentication instead of passwords — eliminates many traditional credential attack surfaces but introduces new ones specific to its implementation. Passkey synchronization security: consumer passkeys synchronized through iCloud Keychain, Google Password Manager, or Windows Hello are protected by the platform vendor's cloud security — a compromise of the synchronization account provides access to all synchronized passkeys. Device compromise: a passkey is only as secure as the device that stores it; malware with access to the Windows Hello for Business credential provider can potentially extract or abuse the credential. Enrollment security: enrolling a new passkey typically requires less verification than the original identity proofing, creating an enrollment path that attackers target. The absence of passwords eliminates one attack surface while the implementation details of the specific passwordless mechanism determine what new surfaces are introduced.
— Enterprise Identity Security Architecture
Comprehensive identity protection for regulated organizations
Tier 0 asset protection: securing the most privileged accounts
Tier 0 assets are the accounts and systems that, if compromised, provide complete control over the entire Active Directory environment: domain controllers, the KRBTGT account, enterprise administrator accounts, accounts with DCSync privileges, and systems with unconstrained Kerberos delegation. Tier 0 protection requires controls proportionate to this blast radius. Administrative access to Tier 0 assets must occur exclusively from Tier 0 Privileged Access Workstations — machines with no internet access, no email, no productivity applications, running a hardened OS image, with local administrator accounts that are unique and not shared with any other system. Tier 0 account credentials must never be used on any non-Tier-0 system — a domain administrator account that is also used for email is a credential that email-borne malware can harvest. The KRBTGT account password must be rotated proactively — not just in response to known compromise — and must be rotated twice (once to establish a new current key, once to expire the previous key) to invalidate any forged tickets.
Identity threat detection: detecting active directory attacks
Detecting Active Directory attacks requires monitoring of specific event log patterns that are generated during attack execution. DCSync detection: replication requests to domain controllers from non-DC sources generate event ID 4662 with specific access mask values; monitoring for 4662 events from non-DC workstations in the security audit log on domain controllers provides reliable DCSync detection. Golden Ticket detection: forged Golden Tickets have several observable characteristics — invalid or non-existent user accounts in tickets, Ticket Granting Ticket (TGT) lifetimes that exceed the domain maximum, and ticket attribute values inconsistent with normal domain policy. Microsoft Entra ID Protection and Microsoft Defender for Identity (MDI) provide automated detection for many AD attack patterns including Pass-the-Hash, Pass-the-Ticket, DCSync, and Kerberoasting. MDI requires deployment of sensors on each domain controller and an understanding of the alert tuning required to reduce false positives in specific organizational environments.
Identity governance at advanced maturity: automated lifecycle and SoD
Advanced identity governance moves from periodic manual access review to continuous automated governance driven by authoritative source data. The authoritative source model: HR system is the authoritative source for employee identity and role; when an employee's role changes in HR, automated provisioning workflows update their access across all connected systems within hours, not days. Access policies are defined at the role level rather than the individual level — when a new employee is hired into a role, they receive exactly the access that role requires through automated provisioning, not through manual copying of a colleague's access. Separation of duties constraints are enforced at the policy layer: an employee cannot be granted both a payment approval role and a payment execution role simultaneously, with automated detection and alert when SoD violations are attempted. Access recertification frequency is inversely proportional to access risk: daily automated review for privileged access changes, quarterly human review for high-access roles, annual review for standard access.
Zero trust identity: continuous evaluation and context-aware access
Zero trust identity implementation at maturity goes beyond requiring MFA at login to continuously evaluating the risk of active sessions and adjusting access in real time. Session risk signals that trigger re-authentication or access reduction: network location change (user appears to have moved from corporate network to Tor exit node during an active session), anomalous data access patterns (user accessing a volume of records 50x their baseline during the current session), application behavior anomaly (application making API calls inconsistent with the user's declared activity), and threat intelligence signals (the user's IP address appears in an active threat campaign report during the session). Zero trust identity platforms — Microsoft Entra ID with CAE, Okta ThreatInsight, Ping Identity — implement portions of this continuous evaluation model. Full implementation requires integration between the identity platform, SIEM, UEBA, and endpoint security tools to share session risk signals in real time.
— Authentication Forensics
Investigating authentication-related incidents
Active Directory forensics: timeline reconstruction
Active Directory forensics reconstructs the timeline of attacker activities within the directory environment following a compromise. Key data sources: the Security Event Log on each domain controller (Event IDs 4624/4625 for logon events, 4720/4722/4723/4724/4726/4728/4732/4756 for account and group modifications, 4663/4662 for object access and replication), the Active Directory audit log (changes to user accounts, group memberships, and computer accounts), the Security Information and Event Management platform's aggregated view of these events across all domain controllers, and NTDS.dit forensics using offline analysis tools to determine the state of the directory at specific points in time. The forensic timeline establishes: when the attacker first authenticated to a domain controller, when privileged accounts were added to sensitive groups, when DCSync was executed, and what accounts and hashes were extracted. This timeline drives the scope of credential rotation required for complete eradication.
Authentication log analysis for M365 investigations
Microsoft 365 authentication investigations require analysis of the Unified Audit Log (UAL), the Microsoft Entra ID sign-in logs, and the Microsoft Entra ID audit logs. The sign-in log provides: the user, timestamp, IP address, location, device, application accessed, MFA method used (or not used), and conditional access policy outcome for every authentication event. The audit log provides: changes to user accounts, application registrations, conditional access policies, and directory settings. UAL forensics for account compromise investigations: timeline the sequence from initial compromise (the first successful authentication from an unusual location) through data access (email item access, SharePoint file access, Teams message retrieval) to persistence establishment (inbox rule creation, OAuth application consent, new device registration). Microsoft Graph API enables automated extraction of these logs at scale, supporting investigation of multiple compromised accounts simultaneously.
Credential hygiene assessment: finding gaps before attackers do
Proactive credential hygiene assessment identifies the vulnerabilities that attackers find during post-exploitation before they are exploited. Assessment activities: password hash extraction and offline cracking (using the same tools attackers use — Hashcat — to determine how many domain user accounts have passwords that appear in common wordlists, measuring crack rate against a defined wordlist without using production credentials), privileged account audit (identifying all accounts with Domain Admin, Enterprise Admin, and other privileged group membership, including nested group membership that may not be visible in the direct member list), service account review (identifying service accounts with non-expiring passwords, high privilege levels, and SPNs registered for Kerberoasting), and delegation audit (identifying all accounts and computers with unconstrained or constrained delegation configured, assessing whether each delegation is necessary and appropriately scoped). Each finding drives specific remediation: password rotation for crackable accounts, privilege reduction for over-privileged accounts, gMSA migration for service accounts, and delegation removal or scoping for delegation findings.
Incident response for credential compromise: comprehensive rotation
Credential compromise incident response requires comprehensive rotation that addresses every credential exposed during the compromise — partial rotation leaves the attacker with residual access. The rotation scope is determined by forensic analysis of what credentials the attacker could have accessed: if DCSync was executed, all domain account hashes were exposed and all passwords must be rotated; if LSASS was accessed on specific systems, the accounts authenticated to those systems are exposed; if a specific user's credentials were phished, that user's password and all MFA factors must be rotated. KRBTGT rotation procedure: rotate the KRBTGT password once (creating a new current key while the old key remains temporarily valid), wait for replication across all domain controllers, then rotate again (invalidating the old key). This two-rotation process prevents disruption to active Kerberos sessions while ensuring that any forged Golden Tickets using the old hash are invalidated after the second rotation. Force all users to re-authenticate after password rotation by revoking all active sessions.
— Authentication Program Governance
Managing identity security as a program, not a project
Authentication risk register and exception management
A formal authentication risk register documents every deviation from authentication standards — MFA exceptions, legacy authentication approvals, service account password exceptions, privileged access policy waivers — with the business justification, risk owner, compensating controls, and review date for each. The register transforms implicit risk acceptance (IT knows about the exception but it is not formally documented) into explicit risk acceptance with accountability. Regulators examining authentication compliance expect to find either full compliance or documented exceptions with compensating controls — undocumented exceptions are the worst outcome from an examination perspective, because they indicate that the organization is unaware of its own gaps. The risk register drives the exception review cycle: each exception must be reviewed at its defined interval (typically quarterly for high-risk exceptions, semi-annually for lower-risk), with the risk owner confirming that the exception is still necessary and the compensating controls are still in place.
Authentication technology roadmap: planning the evolution
Authentication technology roadmap planning requires balancing current capability with future direction across a 3-5 year horizon. Near-term (12-18 months): complete MFA deployment for all access paths, disable legacy authentication protocols, implement smart lockout and HIBP breached password screening, deploy Microsoft Defender for Identity or equivalent. Medium-term (18-36 months): deploy FIDO2 phishing-resistant MFA for privileged access, implement PIM for all elevated roles, complete identity governance automation for joiner-mover-leaver processes, deploy UEBA for behavioral authentication monitoring. Long-term (36-60 months): achieve passwordless authentication for all standard users, implement CAE for all critical applications, complete Tier 0 PAW deployment, achieve zero trust identity maturity with continuous session risk evaluation. The roadmap should be reviewed annually, updated to reflect regulatory developments, and presented to executive leadership as part of the annual security program review.
Regulatory compliance for authentication: NY DFS examination readiness
NY DFS 500.12 examination readiness for authentication requires documentation that demonstrates: every access path to covered systems has MFA deployed, MFA is technically enforced (not just policy-required), legacy authentication is disabled for all applicable protocols, privileged access management controls are in place, and the CISO can certify compliance accurately. Evidence that supports examination: conditional access policy exports showing MFA requirements and enforcement scope, authentication logs showing MFA was challenged and satisfied for sample authentication events, legacy authentication audit reports showing zero successful legacy authentications in the past 90 days, PIM activation records showing privileged access is time-limited and logged, and the authentication exception register showing all deviations are documented and reviewed. Examination preparation should include a mock examination exercise conducted 3-6 months before the anticipated examination date, using the same evidence-gathering process a regulator would use to independently verify compliance.
Building the business case for identity investment
Identity security investment justifications must connect to business outcomes that leadership and boards can evaluate. The incident cost framework: calculate the fully-loaded cost of a single credential compromise incident — external IR firm engagement ($50,000-$150,000), business interruption, investigation, notification and remediation, potential regulatory penalties, and reputational impact — and compare against the cost of controls that would have prevented it. The regulatory framework: NY DFS 500.12 compliance is a legal obligation; non-compliance exposes the organization to regulatory penalties and examination scrutiny. The operational efficiency framework: FIDO2 passwordless authentication and SSO reduce per-authentication friction for users; password reset help desk volume (typically 20-30% of IT help desk tickets) is eliminated; account lockout operational burden decreases. The insurance premium framework: demonstrating strong identity security controls to cyber insurers through documented MFA deployment, exception registers, and MDR coverage can improve insurance terms and reduce premium costs relative to less mature programs.
— Summary
Key concepts from this module
DCSync, Golden Ticket, NTLM relay, OAuth token theft, and passwordless pitfalls target different identity layers
DCSync extracts KRBTGT hash enabling Golden Tickets valid for any user with any privilege. NTLM relay uses coerced authentication to impersonate without cracking hashes. OAuth refresh token theft provides persistent access; PRT theft provides comprehensive M365 SSO access. Passwordless security depends on implementation: sync account security, device security, and enrollment verification all matter.
Tier 0 protection requires PAW-only access, unique credentials, and KRBTGT double-rotation
Domain controllers, KRBTGT, enterprise admins, DCSync-privileged accounts, and unconstrained delegation systems are Tier 0. Administrative access only from Tier 0 PAWs with no internet or email. KRBTGT rotation twice to invalidate existing Golden Tickets. MDI with DC sensors detects DCSync, Pass-the-Hash, Kerberoasting, and Golden Ticket patterns.
Authentication forensics: DC event IDs 4662/4624, M365 UAL sign-in logs, proactive credential hygiene assessment
DCSync detection via Event ID 4662 from non-DC sources. M365 compromise timeline from sign-in logs through UAL data access to persistence events. Proactive assessment uses same Hashcat tools attackers use to measure crackable account rate. Credential rotation scope determined by forensic analysis of attacker access — KRBTGT rotation requires two sequential rotations.
Authentication risk register documents all exceptions; roadmap plans 3-5 year evolution; DFS examination requires technical enforcement evidence
Formal risk register transforms implicit to explicit risk acceptance with review cycles. Technology roadmap: near-term (all MFA, legacy disabled), medium-term (FIDO2, PIM, JML automation), long-term (passwordless, CAE, Tier 0 PAW). Examination evidence: conditional access exports, authentication logs, legacy auth audit, PIM records, exception register.
