— What Is Encryption
How encryption scrambles data so only authorized parties can read it
Encryption converts readable data into scrambled ciphertext
Encryption is the process of converting readable data — called plaintext — into an unreadable format called ciphertext, using a mathematical algorithm and a key. Without the correct key, the ciphertext is meaningless to anyone who intercepts it. This is the fundamental mechanism behind nearly all data security: rather than trying to prevent every possible form of access, encryption ensures that even if an attacker obtains the data, they cannot read it. Modern encryption algorithms such as AES-256 are strong enough that brute-forcing the key is computationally infeasible — an attacker cannot guess their way in. The strength of the protection comes from the algorithm and the secrecy of the key, not from obscurity about how encryption works.
The key is what controls access — protecting it is as important as the encryption itself
An encrypted file or drive is only as secure as the key used to encrypt it. The key is the secret that unlocks the ciphertext and converts it back into readable plaintext — a process called decryption. If an attacker obtains the key, encryption provides no protection at all. This is why key management — how keys are generated, stored, distributed, and eventually retired — is treated as a critical security function. A strong encryption algorithm paired with a weak or exposed key is no protection. Common failures include storing the key in the same place as the encrypted data, using a weak passphrase that can be guessed, or sharing the key insecurely. The security of encrypted data is therefore a function of both the algorithm and the protection of the key.
Data must be protected in two states: at rest and in transit
Data exists in two states that require different encryption approaches. Data at rest is data stored on a device, server, or storage system — a laptop hard drive, a database server, a cloud storage bucket. Encryption at rest protects data if the storage medium is physically stolen or the system is accessed by an unauthorized user. Data in transit is data moving across a network — an email being sent, a file being uploaded, a web request being made. Encryption in transit protects data from being intercepted as it moves between systems. Both states require encryption. An organization that encrypts its databases but sends data over unencrypted connections has left one attack surface open. An organization that encrypts network traffic but stores data unencrypted has left another.
Encryption protects confidentiality — it does not verify identity
Encryption and authentication are frequently confused, but they address different security problems. Encryption protects confidentiality — it ensures that data cannot be read by unauthorized parties. Authentication addresses identity — it verifies that the party claiming to be Alice actually is Alice. An encrypted channel can carry a message from an impersonator, because the channel protects the content but not the identity of the sender. This is why encryption is typically combined with digital signatures and certificate validation — to ensure not only that the content is protected, but that it actually comes from who it claims to come from. HTTPS, for example, uses encryption to protect the content of the connection and certificates to verify that the server is actually the organization it claims to be. Both mechanisms together provide a more complete security guarantee.
— Types of Encryption
Symmetric, asymmetric, TLS, and end-to-end encryption explained
Symmetric encryption uses one shared key for both encrypting and decrypting
Symmetric encryption uses a single key that both the sender and the receiver must possess. The sender encrypts the data with the key; the receiver uses the same key to decrypt it. The primary strength of symmetric encryption is speed — algorithms like AES (Advanced Encryption Standard) are extremely fast and efficient, making them practical for encrypting large amounts of data such as entire hard drives, database contents, or file archives. The primary challenge is key distribution: both parties must securely obtain the same key before communication can begin. If the key is intercepted during distribution, the encryption is compromised. Symmetric encryption is used extensively for data at rest — full-disk encryption, encrypted backups, and database encryption all typically use AES-256.
Asymmetric encryption uses a key pair: a public key to encrypt, a private key to decrypt
Asymmetric encryption solves the key distribution problem by using two mathematically related keys: a public key and a private key. The public key can be shared freely with anyone — it is used to encrypt data intended for the key holder. The private key is kept secret and is the only key that can decrypt what the public key encrypted. This means someone can send you an encrypted message without you ever sharing a secret with them — they simply use your public key. The receiver's private key is the only way to decrypt it. Asymmetric algorithms like RSA are slower than symmetric algorithms, so in practice they are typically used to encrypt the symmetric key itself, which is then used for the bulk of the data. This hybrid approach — asymmetric for key exchange, symmetric for data — is the foundation of TLS and HTTPS.
TLS is the protocol that encrypts web traffic — HTTPS means the channel is encrypted, not that the site is trustworthy
Transport Layer Security (TLS) is the protocol responsible for encrypting web traffic, email in transit, and many other forms of network communication. When a browser connects to a site over HTTPS, TLS is establishing an encrypted channel and validating a server certificate — a digital credential that proves the server is operated by the organization it claims to be. The padlock icon in a browser address bar indicates that the connection is encrypted and the certificate validated, which means the communication cannot be intercepted in transit. It does not mean the site is safe, legitimate, or that the organization operating it can be trusted. Phishing sites routinely use HTTPS and display the padlock. The padlock means your connection to the site is private; it says nothing about whether the site itself is malicious.
End-to-end encryption ensures only the communicating endpoints can read the content — not even the service provider
End-to-end encryption (E2EE) is a form of encryption in which the data is encrypted on the sender's device and only decrypted on the recipient's device, with no decryptable form existing on the servers in between. The service that delivers the message holds only ciphertext it cannot read. This is the encryption model used by messaging applications like Signal and, for specific message types, WhatsApp. The practical implication is that even if the service provider receives a legal order to produce the content of communications, it cannot comply — it has no key. E2EE provides strong privacy guarantees but introduces key management challenges: if a user loses their device or credentials without a backup, they may permanently lose access to encrypted content. Organizations evaluating E2EE applications should understand both the privacy benefits and the data recovery implications.
— Encryption in Practice
Full-disk encryption, file encryption, email, and cloud storage
Full-disk encryption protects a device if it is lost or stolen — but only when it is powered off
Full-disk encryption (FDE) encrypts the entire contents of a storage drive so that all data is stored in encrypted form. On Windows, this is implemented through BitLocker; on macOS, through FileVault. When FDE is enabled and a device is powered off, the drive contents are encrypted and unreadable to anyone who does not have the decryption key or credentials. This means a laptop stolen from a bag in an airport — powered off — exposes no readable data, even if the drive is removed and connected to another computer. However, full-disk encryption does not protect a device that is powered on and left unlocked. Once the user has authenticated and the drive is unlocked, the data is accessible to anyone who sits at the keyboard. FDE and strong screen lock policies work together — FDE without an automatic screen lock leaves a critical gap.
File transfers must use encrypted protocols — FTP and plain HTTP transmit data in the clear
When files are transferred over a network, the choice of protocol determines whether the contents can be intercepted. FTP (File Transfer Protocol) and plain HTTP transmit data without encryption — anyone with access to the network path can capture the contents in readable form. This includes not only external attackers but also insiders on shared networks, cloud provider employees with access to network infrastructure, or anyone who has compromised a network device along the path. SFTP (SSH File Transfer Protocol) and FTPS (FTP over TLS) both encrypt the connection and should always be used in place of plain FTP. HTTPS encrypts web-based file transfers. For organizations transmitting client data, financial records, or regulated personal information, using unencrypted transfer protocols is itself a compliance failure under frameworks including NY DFS 500, HIPAA, and PCI DSS.
Email is encrypted in transit by TLS — but TLS alone means your mail provider can still read the content
Most business email is transmitted between mail servers over TLS, which encrypts the connection so the content cannot be intercepted in transit. However, TLS encryption of email in transit does not encrypt the content at rest on the mail server. The email provider stores readable copies of every message. This is sufficient for most business purposes, but for communications that genuinely require confidential handling — legal matters, regulated financial information, sensitive HR communications — content-level email encryption is appropriate. S/MIME (Secure/Multipurpose Internet Mail Extensions) uses digital certificates to encrypt email content end-to-end, so the content is only readable by the intended recipient. PGP (Pretty Good Privacy) provides similar functionality but requires both parties to have exchanged keys. Content encryption must be configured explicitly — it does not happen automatically from TLS alone.
Cloud storage providers encrypt data at rest — but who holds the key determines who can actually access it
Cloud storage services like Microsoft OneDrive, Google Drive, and AWS S3 encrypt stored data by default. This protects against scenarios like a hard drive being physically removed from a data center. However, default provider-managed encryption means the provider controls the encryption keys — they can decrypt the data themselves, comply with government orders to produce it, and access it if their key management systems are compromised. Client-side encryption or customer-managed keys (CMK) are alternatives in which the organization retains control of the keys. Data is encrypted before it leaves the organization's control, or keys are managed in a system the organization controls rather than the provider. For high-sensitivity data, this distinction matters significantly. Understanding which model applies to each data store is a necessary part of any encryption inventory.
— When Encryption Fails
The human and configuration failures that defeat strong cryptography
A strong encryption algorithm with a weak passphrase provides almost no protection
AES-256 encryption cannot be brute-forced in any practical timeframe — an attacker would need computational resources beyond anything that exists or is foreseeable. However, if the encryption key is derived from a passphrase, the attacker does not need to break the algorithm — they only need to guess the passphrase. A passphrase like "password," "123456," or the organization's name can be guessed in seconds using dictionary attacks against common patterns. A passphrase that is twelve random characters or a four-word phrase is far more resistant. This is why password policies for full-disk encryption, encrypted archives, and encrypted backup keys require strong passphrases specifically — the cryptographic strength of the algorithm only matters if the input is also strong. The weakest link in any encryption implementation is the password it relies on.
Sending sensitive data over an unencrypted connection exposes it regardless of how the data is stored
An organization may store its data encrypted, but if that data is transmitted over an unencrypted channel, it is exposed in transit. HTTP transmits all data including usernames, passwords, session tokens, and document contents in plaintext across the network. Any device on the same network — a compromised router, a packet capture tool, a shared Wi-Fi access point — can capture this traffic. This is a particular risk on public Wi-Fi networks where other users share the same connection. The practical implication for employees is that internal portals, applications, and services should always be accessed over HTTPS, and any system still serving traffic over plain HTTP warrants a report to the IT team. Credentials entered on HTTP connections should be treated as potentially compromised and changed.
Full-disk encryption is bypassed entirely when the device is left unlocked and unattended
Full-disk encryption protects data when a device is powered off. Once the user authenticates at startup, the disk is decrypted and accessible. If that user then walks away without locking the screen, the device is a fully accessible, decrypted workstation — anyone who sits at it has the same access the authorized user would have. FDE provides no protection against this scenario. The policy control that addresses it is automatic screen lock: a short idle timeout (two to five minutes) that requires re-authentication to unlock. Employees who leave devices unattended in shared spaces — open office environments, conference rooms, hotel lobbies — without locking the screen are not protected by their FDE, regardless of whether it is enabled and configured correctly. Locking a screen when leaving a device, even briefly, is the habit that makes FDE effective.
Losing the encryption key means permanently losing access to the data — key management is as important as encryption itself
Encryption is only reversible with the correct key. If the key is lost — the passphrase forgotten, the certificate expired and not renewed, the key management system deleted, or the hardware security module failed without backup — the encrypted data is permanently inaccessible. This is not a recoverable situation. There is no back door in well-implemented encryption; that is its strength. For organizations this means that encryption key management must include documented backup procedures, recovery key storage in a secure location separate from the encrypted data, and regular verification that recovery keys actually work. BitLocker recovery keys, for example, should be escrowed to Active Directory or a management platform. An encrypted backup that cannot be decrypted because the key is unavailable provides no recovery capability at all. Key management is the operational discipline that makes encryption sustainable.
— Summary
Four key encryption principles every employee should know
Encryption must be applied in both states: at rest and in transit
Data that is encrypted at rest but transmitted unencrypted is exposed in transit. Data encrypted in transit but stored unencrypted is exposed at rest. Both states require encryption, and a gap in either represents a genuine risk. In practice: use full-disk encryption on all endpoints, ensure cloud storage services encrypt data at rest, confirm that all network communications use TLS, and avoid protocols like FTP and HTTP that transmit data in plaintext.
HTTPS means the channel is encrypted — it does not mean the destination is trustworthy
The padlock in a browser address bar confirms that the connection is encrypted and the server certificate validated. It does not confirm that the site is legitimate, safe, or operated by who it claims to be. Phishing sites routinely use HTTPS. The correct response to seeing a padlock is to also verify the domain name — confirm you are on the organization's actual domain and not a lookalike. HTTPS answers one question: is my connection private? It does not answer: should I trust this site?
Full-disk encryption protects a powered-off device — screen lock is what protects a powered-on one
FDE and screen lock are complementary, not interchangeable. Enable BitLocker or FileVault on every organizational device. Set automatic screen lock to activate after two to five minutes of idle time. Shut down rather than sleep when leaving devices in risky environments. Store BitLocker recovery keys in the organization's key management system — if the key is lost, the data is lost permanently. These four practices together make FDE effective.
The human layer determines whether encryption works — passphrases, key backup, and protocol discipline
Strong encryption algorithms are essentially unbreakable when implemented correctly. The failures that defeat encryption happen at the human and configuration layer: weak passphrases, unencrypted protocols chosen out of habit, keys left without backups, and devices left unlocked. The employee's role is to use strong passphrases wherever encryption is passphrase-protected, to use only encrypted protocols for sensitive communications, to lock screens when stepping away from devices, and to report any system that appears to be operating over unencrypted channels.
