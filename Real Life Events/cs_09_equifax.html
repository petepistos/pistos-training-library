— What Happened
2017 — a missing patch exposed 147 million Americans
The largest consumer data breach in history at that time
In September 2017, Equifax — one of the three major US credit bureaus — disclosed that attackers had accessed the personal information of approximately 147 million Americans. The stolen data included Social Security numbers, birth dates, home addresses, driver's license numbers, and approximately 209,000 credit card numbers. This was not data that consumers had chosen to share with Equifax. It was data that Equifax had collected as part of its credit reporting function — data that Americans had no ability to opt out of providing. The breach exposed the personal information of nearly half the US population.
The patch that was not applied
The entry point was a vulnerability in Apache Struts, a widely-used open-source web application framework. The vulnerability — CVE-2017-5638 — was publicly disclosed and a critical patch was released by the Apache Software Foundation on March 7, 2017. Equifax's security team was notified of the vulnerability and the available patch. Equifax did not apply the patch for 78 days. On May 13, 2017, attackers exploited the unpatched vulnerability and established access to Equifax's systems. The breach that followed was entirely preventable by applying a publicly available patch before attackers used the vulnerability.
76 days inside the network, undetected
From May 13 through July 30, 2017 — 76 days — attackers moved through Equifax's network, identified and queried databases containing consumer data, and exfiltrated 147 million records. During this period, Equifax's security monitoring was substantially hampered by an expired SSL certificate that had disabled encrypted traffic inspection for 19 months. The attackers' traffic was therefore invisible to Equifax's intrusion detection system. It was only when the expired certificate was finally renewed on July 29 and traffic inspection resumed that the anomalous activity was detected.
The disclosure and its immediate consequences
Equifax publicly disclosed the breach on September 7, 2017 — more than six weeks after internal discovery on July 29. The disclosure triggered immediate regulatory and congressional attention. CEO Richard Smith, CIO David Webb, and CSO Susan Mauldin all left the company within weeks of the disclosure. The FTC, CFPB, and state attorneys general across 48 states launched investigations. Congressional hearings featured the spectacle of Richard Smith testifying that the breach was attributable to one employee who failed to implement a patch notification. The settlement, reached in 2019, was $575 million — the largest data breach settlement in FTC history.
— How It Happened
A known vulnerability, an expired certificate, and 76 days undetected
Step 1: Apache Struts vulnerability exploited
CVE-2017-5638 is a remote code execution vulnerability in Apache Struts. It allows an attacker to execute arbitrary commands on a server by sending a specially crafted HTTP request in the Content-Type header. No authentication is required. The vulnerability affects any server running the unpatched version of Apache Struts 2. Equifax's consumer dispute portal was running the unpatched version. On May 13, 2017, attackers sent a crafted request to the portal, which executed code on the server. The attackers had gained a shell — remote command execution — on Equifax's web server.
Step 2: Lateral movement through Equifax's network
From the web server, the attackers moved laterally through Equifax's network. They identified credentials, moved from system to system, and ultimately located the databases containing consumer data. This lateral movement occurred across 76 days, during which Equifax's monitoring was unable to observe the attackers' encrypted communications due to the expired SSL certificate that had disabled traffic inspection.
Step 3: 48 databases queried over 400 times
The attackers identified 48 databases containing consumer data and queried them more than 400 times. They exfiltrated the data in encrypted chunks to prevent detection. The 147 million consumer records they accessed represented essentially a complete copy of Equifax's consumer data — names, Social Security numbers, birth dates, addresses, and credit card numbers for a population the size of nearly half the United States.
Step 4: An expired certificate ended the invisibility
On July 29, 2017, Equifax renewed an SSL certificate that had been expired for 19 months. When the certificate was renewed, encrypted traffic inspection resumed. The intrusion detection system, now able to see the traffic it had been blind to for 19 months, detected the anomalous activity. The attackers had been inside the network for 76 days. Equifax's security team discovered the breach on July 29, confirmed it, and did not publicly disclose it until September 7 — 40 days later.
— Root Cause
Four compounding failures that exposed half the US population
Failure 1: Critical patch not applied for 78 days
The Apache Struts patch was available on March 7, 2017. Equifax's security team was notified. The patch was not applied for 78 days. A patch management program with accountability for critical patch application — a defined maximum timeline, a tracking system identifying unpatched systems, and escalation when patches are not applied within that timeline — would have closed the vulnerability before attackers exploited it on May 13. The breach was entirely preventable by a widely available fix that Equifax had been told about and did not apply.
Failure 2: Expired SSL certificate disabled traffic inspection
Equifax's intrusion detection system depended on decrypting and inspecting SSL/TLS encrypted traffic. The SSL certificate used for this inspection had expired 19 months before the breach. The expired certificate meant that encrypted traffic inspection was disabled, making the attackers' traffic invisible to the IDS. Certificate lifecycle management — maintaining an inventory of all certificates with automated alerts when expiration approaches — is the control that prevents this failure. An expired certificate in an organization Equifax's size is an organizational process failure, not a technical inevitability.
Failure 3: No data minimization — storing more than necessary
The 147 million records Equifax stored represented data that the company had collected over time, including data that may no longer have been necessary for its credit reporting function. Data minimization — retaining only the data necessary for defined business purposes and deleting data that is no longer needed — limits the scope of any breach. If Equifax had only retained the data it needed, the breach would have been smaller. Data minimization is both a security principle and a regulatory requirement under most modern privacy regulations.
Failure 4: Web server had direct access to database systems
The Apache Struts vulnerability gave attackers code execution on a web server. From that web server, they were able to reach 48 separate databases containing consumer data. A properly segmented application architecture places web servers in a DMZ with no direct access to database servers. Database servers should be in a protected zone accessible only from application servers with a defined business need to access them. The web server compromise gave attackers a path to the database tier because no architectural boundary blocked that path.
— The Loss
$700M+ in settlements and the permanent identity theft risk for 147M people
Financial: $700M+ in settlements and legal costs
The FTC, CFPB, and attorneys general from 48 states and territories reached a settlement with Equifax in July 2019 for $575 million, structured as $300 million for consumer credit monitoring, $175 million to state AGs, and $100 million in FTC civil penalties. This was the largest data breach settlement in FTC history. Total costs including legal fees, remediation, and additional class action settlements exceeded $700 million. Equifax also spent hundreds of millions of dollars improving its cybersecurity infrastructure in the years following the breach.
Consumer harm: permanent identity theft risk for 147M people
The specific data stolen from Equifax — Social Security numbers combined with birth dates, addresses, and credit card numbers — is precisely the combination required to open fraudulent credit accounts, file fraudulent tax returns, and commit a wide range of identity fraud. Unlike a stolen credit card number, which can be canceled and replaced, a Social Security number cannot be changed. The 147 million people whose SSNs were stolen face an elevated identity theft risk that is effectively permanent. Years after the breach, the stolen data continues to circulate and be used for fraud.
Leadership consequences: CEO, CIO, and CSO all resigned
CEO Richard Smith retired within days of the disclosure. CIO David Webb and CSO Susan Mauldin both left the company shortly after. The departure of three executives, including the CEO, within weeks of a security breach was unusual enough to attract significant commentary. Smith's congressional testimony — in which he attributed the breach to one employee who failed to relay a patch notification — drew criticism for its attempt to locate responsibility in a single individual rather than in the organizational processes that should have ensured patch application. The accountability norms established by the Equifax case have influenced expectations for executive cybersecurity accountability since.
Industry lesson: data collectors bear responsibility for data protection
Equifax collected data on 147 million Americans who had no ability to opt out and no awareness that their data was held there. The breach highlighted the specific risk posed by organizations that aggregate large volumes of personal data as a function of their business model. The principle that organizations should only collect and retain data they need for defined business purposes — and should protect what they collect with security commensurate to its sensitivity — applies with particular force to credit bureaus, insurance companies, healthcare providers, and any other organization that collects sensitive personal data as a core business function.
— Lessons Learned
What Equifax teaches about patching, certificates, and data minimization
A known vulnerability with an available patch is a choice to stay at risk
CVE-2017-5638 had a patch available 78 days before it was exploited. The breach was preventable. Every organization must have a patch management program with defined timelines, tracked compliance, and accountability for gaps. Critical patches — those addressing remotely exploitable vulnerabilities — require priority treatment and should never remain undeployed for weeks after release.
An expired certificate that disables security inspection is a critical failure
Equifax's IDS was blind to the attackers for 76 days because an SSL certificate had expired 19 months earlier. Certificate lifecycle management — an inventory of all certificates with automated expiration alerts — is a basic operational control. When a certificate used for traffic inspection expires, the organization loses visibility into encrypted traffic. That loss of visibility must be treated as a security incident, not a routine administrative task.
Data minimization limits the scope of every breach
You cannot lose data you do not have. Data minimization — retaining only what is necessary for defined business purposes, and deleting what is not — limits the scope of any breach. If Equifax had only retained current, necessary data, the breach would have been smaller. Reviewing retention policies and deleting unnecessary data is both a security practice and a regulatory requirement.
Web servers should never have direct access to database servers
A compromise of a web server should not give access to backend databases. The application architecture should place web servers in a DMZ with no direct database connectivity. Databases should be accessible only from specific, authorized application servers. This segmentation means that even a complete compromise of the web tier does not automatically grant access to the data tier.
