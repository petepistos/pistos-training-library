— Security Architecture Principles
The frameworks and principles that guide secure system design
Defense in depth: why no single control is sufficient
Defense in depth is the architectural principle that security controls must be layered so that the failure of any single control does not result in a complete security failure. A firewall blocking external access is useful, but if it is the only control, a single misconfiguration or exploit defeats all security. When the firewall is complemented by network segmentation, endpoint detection, authentication controls, and monitoring, an attacker must defeat multiple independent controls sequentially to reach their objective. Each layer should be independently designed and independently operated where possible — controls implemented in the same platform or by the same team can fail simultaneously through shared vulnerabilities or shared configuration errors. Defense in depth is not about redundancy of the same control type; it is about diversity of control types covering the same threat from different angles.
Least privilege: minimum access required for each function
The principle of least privilege holds that every user, process, and system component should have access to only the resources and capabilities required for its specific function, and no more. In practice, least privilege is violated constantly through over-provisioning: giving new employees broad access because scoping precisely is difficult, granting Domain Admin to service accounts because it is easier than determining the minimum permissions needed, and allowing applications to run as SYSTEM because the developer did not restrict the execution context. Each over-provisioned permission is attack surface that an adversary can exploit. An application that runs as a standard user can only do damage within that user's permissions; the same application running as SYSTEM can modify the operating system, disable security tools, and access any file on the disk. Implementing least privilege requires explicit effort — the default in most systems is to grant broad access and restrict specific actions, when the correct approach is the inverse.
Fail secure: what happens when controls fail
Fail secure (also called fail closed or fail safe depending on the domain) is the design principle that when a security control fails, it should fail in the state that maintains security rather than the state that maintains availability. A firewall that fails by blocking all traffic is fail secure — connections cannot pass, but neither can attackers. A firewall that fails by allowing all traffic is fail open — the system remains available but completely unprotected. Authentication systems that fail by granting access rather than denying access are fail open and represent a significant security risk. Network access controls that fail by placing devices on the production network rather than the remediation VLAN are fail open. Designing for fail secure requires explicitly considering failure modes for every control and ensuring that the failure state is the secure one, accepting that availability may be impacted during failure rather than security.
Separation of duties: preventing single points of control
Separation of duties is the architectural principle that critical functions should require multiple independent actors to complete, so that no single person or system can unilaterally perform the function. In finance, no single individual should be able to both authorize and execute a wire transfer. In IT operations, no single administrator should be able to both make a production change and approve it in the change management system. In security, no single person should be able to both create security exceptions and approve them. Separation of duties limits insider threat (a malicious actor cannot act alone), limits the impact of compromised credentials (an attacker who compromises one account cannot complete a two-person function), and creates audit trails (each step in a multi-person process is logged independently). Implementing separation of duties requires process redesign and tool support — technical enforcement is more reliable than policy-only controls.
— Secure Architecture Patterns
Common patterns for building secure systems
Zero trust network architecture: verify every request
Zero trust architecture abandons the perimeter security model's implicit trust for internal network traffic. In a zero trust model, every access request — regardless of whether it comes from inside the corporate network, a VPN, or the open internet — must be authenticated, authorized, and continuously evaluated before access is granted. The architectural components of zero trust include: a strong identity foundation (every user and device has a verified identity, not just an IP address), device health verification (managed, patched, compliant devices have different access rights than unmanaged devices), application-level access controls (users access specific applications, not network segments), and micro-segmentation (east-west traffic between systems is restricted, not assumed trusted). Zero trust is an architectural direction, not a single product — organizations implement it progressively across the identity, device, network, application, and data dimensions.
API security architecture: protecting service interfaces
Modern applications are built on APIs — interfaces that allow systems to communicate. APIs represent a major attack surface because they expose application logic and data access to any caller who can reach them. Secure API architecture requires: authentication for every API call (API keys, OAuth tokens, or mutual TLS certificates), authorization checks that verify the caller is permitted to perform the specific action on the specific resource, input validation that rejects malformed or unexpected input before it reaches business logic, rate limiting that prevents abuse through excessive API calls, logging of all API calls for security monitoring, and API gateway infrastructure that enforces these controls centrally rather than depending on individual API implementations. APIs that are publicly accessible require all of these controls; internal-only APIs still require authentication and authorization because internal network access cannot be assumed trustworthy in a zero trust architecture.
Cryptographic architecture: selecting and implementing encryption
Cryptographic architecture encompasses all decisions about how data is protected at rest, in transit, and in use. Key decisions include: algorithm selection (AES-256 for symmetric encryption, RSA-2048 or higher for asymmetric, SHA-256 or higher for hashing — and post-quantum-ready alternatives for long-lived data), key management (where keys are stored, how they are generated, when they are rotated, and who can access them), certificate management (PKI hierarchy design, certificate validity periods, certificate lifecycle automation), and protocol selection (TLS 1.2 minimum, TLS 1.3 preferred, with secure cipher suite configuration). Cryptographic architecture failures — using deprecated algorithms, hardcoding encryption keys in application code, neglecting certificate rotation — are common sources of security vulnerabilities. Hardware security modules (HSMs) provide tamper-resistant key storage for the most sensitive cryptographic material.
Resilience architecture: designing for availability under attack
Security architecture must consider availability as a security property, not just confidentiality and integrity. Resilience architecture designs systems to continue operating under attack or failure conditions. Key patterns include: redundancy (multiple independent instances of critical components so that the failure of one does not cause total outage), geographic distribution (components spread across locations so that a physical event or regional disruption does not affect all instances), graceful degradation (the ability to provide reduced functionality rather than complete failure when some components are unavailable), and chaos engineering (deliberately introducing failures in controlled conditions to verify that resilience designs actually work). Distributed denial of service (DDoS) resilience requires upstream scrubbing, CDN distribution, and rate limiting architecture designed for high-volume attack traffic. Backup and recovery architecture must ensure that recovery time and recovery point objectives are achievable under realistic incident conditions, not just in vendor documentation.
— Threat Modeling
Systematic analysis of what can go wrong and how to prevent it
STRIDE: categorizing threats systematically
STRIDE is a threat categorization framework developed at Microsoft that provides a systematic vocabulary for identifying security threats. Spoofing — impersonating another user, system, or component to gain unauthorized access. Tampering — modifying data or code without authorization, including data in transit, at rest, or in processing. Repudiation — denying having performed an action, either maliciously or through lack of audit trails. Information disclosure — exposing data to unauthorized parties, whether through direct access, inference, or side channels. Denial of service — preventing legitimate access by exhausting resources or disrupting availability. Elevation of privilege — gaining capabilities or permissions beyond what was granted, either through vulnerability exploitation or misconfiguration. Applying STRIDE systematically to each component, data flow, and trust boundary in a system architecture ensures comprehensive threat identification rather than ad hoc enumeration of the threats that come to mind first.
Attack trees: modeling attacker paths to objectives
Attack trees model the paths an attacker might take to achieve a specific objective, structured as a tree with the attacker's goal at the root and the specific actions required to achieve it as branches and leaves. Each node in the tree represents a subgoal or action; branches represent alternative paths (OR nodes, where any branch achieves the goal) or required combinations (AND nodes, where all branches must succeed). Attack trees enable systematic enumeration of attack paths, analysis of which paths are most likely based on attacker capability and motivation, and identification of where controls would most effectively block the most paths. A tree for 'gain administrative access to the billing system' might branch into 'steal administrator credentials,' 'exploit a vulnerability in the billing application,' and 'obtain physical access to the server room' — each with its own sub-branches that reveal the specific actions each path requires.
Data flow diagramming: identifying trust boundaries and data exposure
Data flow diagrams (DFDs) document how data moves through a system — where it originates, how it is processed, where it is stored, and what external entities interact with it. Trust boundaries in a DFD mark the transitions between components with different trust levels: the boundary between the internet and the DMZ, between the web tier and the application tier, between the application tier and the database. Data flows that cross trust boundaries are the primary focus of threat modeling analysis because they represent points where data transitions between environments with different security controls. DFD threat modeling asks: what happens to data at each trust boundary crossing? Is authentication enforced? Is data validated before processing? Is sensitive data encrypted in transit? Are audit logs capturing the data access? Systematically answering these questions for every trust boundary crossing identifies security controls that are missing or insufficient.
Threat modeling in practice: when and how to apply it
Threat modeling is most valuable when performed early in the design process, before implementation decisions are locked in — changing an architecture to add a missing security control before code is written is far less expensive than retrofitting the same control after deployment. However, threat modeling can be applied to existing systems to identify gaps and prioritize remediation. The practical process: create a data flow diagram of the system or feature being analyzed, identify all trust boundaries, apply STRIDE to each data flow crossing a trust boundary and each component, prioritize identified threats by likelihood and impact, and design mitigations for each significant threat. Threat modeling should be a recurring practice for material changes to systems, not a one-time exercise. The output is a documented list of identified threats and their mitigations — a living document that evolves with the system architecture.
— Security Architecture for Insurance Agencies
Applying security architecture principles to the NY DFS regulatory environment
NY DFS 500 architectural requirements and their implementation
NY DFS 23 NYCRR 500 mandates specific security architectural elements that covered entities must implement. Multi-factor authentication is required for access to critical systems and nonpublic information — architectural implementation requires selecting MFA methods appropriate to each access pattern and ensuring MFA is enforced at the identity layer, not just at individual application login pages. Encryption is required for NPI in transit and at rest — architectural implementation requires a cryptographic inventory identifying where NPI is stored and how it moves, and verifying that encryption is applied at each point. Privileged access management is required — architectural implementation requires separating privileged accounts from standard user accounts, implementing PAM tooling for credential vaulting, and establishing JIT access workflows. Each requirement maps to specific architectural decisions and technical implementations that must be documented and demonstrable during regulatory examination.
Third-party and vendor security architecture
Insurance agencies typically rely on many third-party systems — agency management systems, comparative raters, carrier portals, email platforms, document management systems. Each represents an architectural integration point that may create security risk. The architectural principles for third-party integration include: minimum necessary access (third-party systems should have access only to the data they need to perform their function, not broad access to all organizational data), authentication and access control at the integration boundary (API integrations should use secure authentication methods, not shared passwords), monitoring of third-party access (API calls from third-party systems should be logged and monitored for anomalous patterns), and vendor security assessment before integration (evaluating whether the third party's security controls meet the organization's standards before granting access). NY DFS 500.11 mandates a third-party service provider security policy covering these elements.
Security architecture documentation and evidence
Security architecture for regulated entities must be documentable and evidentiable — regulators expect to see written documentation of architectural decisions, not just implemented systems. Required documentation includes: network architecture diagrams showing all network zones, trust boundaries, and security controls at each boundary; data flow diagrams showing where NPI is stored, processed, and transmitted; cryptographic architecture documentation showing what algorithms are used where and how keys are managed; access control architecture documentation showing how least privilege is implemented and how privileged access is managed; and incident response architecture showing detection, containment, and notification capabilities. This documentation serves multiple purposes: it supports regulatory examination, it enables security assessments to identify gaps, it ensures continuity when key personnel change, and it demonstrates that security architecture is designed, not accidental.
Security architecture review and continuous improvement
Security architecture is not a static design — it must evolve as the threat landscape, technology environment, and regulatory requirements change. Architecture review processes that identify when security review is required (new system deployments, significant changes to existing systems, new integration partners, regulatory changes) ensure that new components meet the same standards as the existing architecture. Security architecture reviews assess whether proposed designs align with security principles, whether identified threats have appropriate mitigations, whether new components introduce gaps in existing controls, and whether documentation is maintained. Post-incident architecture reviews examine whether the incident revealed architectural weaknesses that should be addressed through design changes rather than just operational response. The goal is an architecture that improves continuously rather than one that was designed once and degraded as the environment changed around it.
— Summary
Key concepts from this module
Defense in depth, least privilege, fail secure, and separation of duties are the foundational principles
Defense in depth layers independent controls so no single failure is catastrophic. Least privilege limits attack surface by restricting access to what is needed. Fail secure ensures control failure defaults to the protected state. Separation of duties requires multiple actors for critical functions, limiting insider threat and compromised credential impact.
Architecture patterns: zero trust, API security, cryptographic architecture, and resilience design
Zero trust verifies every request regardless of network location. API security requires authentication, authorization, input validation, and rate limiting at every interface. Cryptographic architecture covers algorithm selection, key management, and protocol configuration. Resilience architecture designs for availability under attack through redundancy, distribution, and graceful degradation.
Threat modeling uses STRIDE, attack trees, and DFDs to systematically identify threats before they are exploited
STRIDE categorizes threats as Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege. Attack trees model paths to attacker objectives. DFDs identify trust boundaries where security controls must be applied. Threat modeling is most valuable early in design but applicable to existing systems for gap identification.
NY DFS 500 requires documented security architecture covering MFA, encryption, PAM, and third-party controls
Each regulatory requirement maps to specific architectural implementations that must be documented and evidentiable. Third-party integrations require minimum access, secure authentication, monitoring, and vendor assessment. Architecture must evolve through formal review processes as systems, threats, and regulations change.
