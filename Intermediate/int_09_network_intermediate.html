— Network Architecture and Segmentation
Designing networks to contain threats and limit lateral movement
Defense in depth: layered network zones and trust boundaries
Defense in depth applied to network architecture means that no single compromised segment provides access to all organizational resources. Traditional flat networks — where any device can reach any other device — allow an attacker who compromises a single workstation to immediately reach every server, every storage system, and every other endpoint. Layered network zones create boundaries that an attacker must cross sequentially: an internet-facing DMZ hosts public-facing services; an internal network hosts user workstations and productivity systems; a server zone hosts application servers with restricted inbound access; a data zone hosts databases and storage with access only from application servers; a management zone hosts administrative infrastructure accessible only from privileged workstations. Each zone boundary is enforced by a firewall or access control list, requiring the attacker to obtain credentials or exploit vulnerabilities to cross each boundary.
Micro-segmentation: east-west traffic control at the workload level
Traditional network segmentation controls north-south traffic — traffic entering and leaving the network perimeter — but does not restrict east-west traffic between systems within the same network segment. Micro-segmentation extends access control to the workload level, applying firewall rules between individual servers, virtual machines, or containers within the same network segment. A web server that should only communicate with a specific application server and a load balancer has firewall rules that prevent it from initiating connections to any other internal resource, even within the same VLAN. This limits the blast radius of a compromised workload: lateral movement requires crossing each micro-segmentation boundary with appropriate credentials or vulnerabilities, rather than having unrestricted access to all systems reachable within the compromised segment. Software-defined networking platforms including VMware NSX, Cisco ACI, and cloud-native security groups implement micro-segmentation at scale.
Zero trust architecture: verify every request regardless of network location
Zero trust architecture abandons the perimeter security model's implicit assumption that traffic from inside the network is trustworthy. In a zero trust model, every access request — regardless of whether it originates from inside the corporate network, from a VPN connection, or from the internet — must be authenticated, authorized, and continuously evaluated. The principles include: verify explicitly (authenticate every user, device, and application for every access request), use least privilege access (grant the minimum access needed for the specific task), and assume breach (design systems assuming that any component may already be compromised, and build detection and containment accordingly). Zero trust does not mean eliminating network controls; it means that network location is not used as an implicit trust signal. Access decisions are based on identity, device health, resource sensitivity, and behavioral signals, not on whether the request comes from inside the firewall.
Network access control: enforcing device compliance at connection time
Network access control (NAC) evaluates device compliance before granting network access. A NAC system intercepts new connections — whether wired, wireless, or VPN — and assesses the connecting device against defined compliance criteria: is the device managed by the organization, is endpoint protection software installed and current, is the operating system at the required patch level, and does the device have required security configurations applied? Devices that meet compliance criteria are placed on the production network. Non-compliant devices are quarantined to a remediation VLAN with limited connectivity — typically only access to update services to achieve compliance — until they meet requirements. Guest devices are placed on a guest network with only internet access and no internal network connectivity. NAC prevents unmanaged, potentially compromised devices from accessing internal resources while allowing legitimate users to remediate compliance issues.
— Firewall and Perimeter Controls
Configuration principles for effective network boundary enforcement
Next-generation firewall capabilities: application awareness and TLS inspection
Next-generation firewalls (NGFW) extend traditional port-based firewall rules with application-layer inspection, user identity awareness, and encrypted traffic analysis. Application identification allows rules based on the actual application generating traffic rather than just the port number — a rule can allow Salesforce traffic on HTTPS while blocking other HTTPS applications from the same users, or restrict which departments can access specific cloud services. User identity integration ties firewall rules to individual users or groups rather than IP addresses, which change with DHCP and remote work. TLS/SSL inspection decrypts HTTPS traffic, inspects the content, and re-encrypts before forwarding, allowing the firewall to identify threats hidden in encrypted sessions. TLS inspection requires certificate trust configuration on endpoints and raises privacy considerations for personal traffic that must be addressed in policy.
Firewall rule hygiene: the accumulation problem and audit practices
Firewall rule sets accumulate over time. Rules are added for new business requirements — a vendor needs access to a specific port, a new application requires inbound connectivity — but rules are rarely removed when the need expires. Over months and years, firewall rule sets grow into hundreds or thousands of rules containing: rules for systems that have been decommissioned, rules for vendors whose contracts have ended, overly broad rules where a specific exception became permanent, shadow rules that are shadowed by earlier rules and never match, and redundant rules that duplicate more specific rules. Periodic firewall rule audits — reviewing every rule against current business justification — are essential security hygiene. Each unexplained or unjustified rule represents a potential access path that an attacker or insider could exploit. Firewall vendor platforms provide rule analysis tools that identify shadowed, unused, and redundant rules to prioritize audit effort.
Egress filtering: controlling outbound traffic
Most firewall configurations focus on inbound traffic — what can reach internal systems from the internet. Egress filtering applies the same rigor to outbound traffic — what internal systems are permitted to send to external destinations. Without egress filtering, a compromised internal system can freely establish C2 communications, exfiltrate data, and reach any external destination. Effective egress filtering: restricts outbound traffic to known-good destination categories (business application categories, update services, approved cloud services), blocks outbound connections to known malicious infrastructure using threat intelligence feeds, limits which internal systems can initiate outbound connections (workstations may need broad access; servers should not be initiating arbitrary outbound connections), and generates alerts for outbound connections to newly registered or unusual domains. Egress filtering does not prevent all C2 and exfiltration but significantly increases the cost of establishing covert communications.
DMZ architecture and exposure minimization
The demilitarized zone (DMZ) hosts services that must be reachable from the internet while isolating them from the internal network. Properly architected DMZ environments: place internet-facing services in the DMZ with firewall rules allowing only the specific ports required for their function, restrict DMZ-to-internal communication to only the specific ports and destinations required for backend connectivity (for example, a web server in the DMZ connects only to a specific application server on the internal network on a specific port, not to the entire internal network), prohibit direct inbound connections from the internet to the internal network (all inbound traffic terminates in the DMZ and is proxied or relayed if internal connectivity is needed), and apply hardened configurations to DMZ systems since they are the most exposed. Internet-facing attack surface minimization — ensuring that only systems that must be accessible from the internet are accessible — is the single most important DMZ security principle.
— Network Monitoring and Threat Detection
Detecting threats through network traffic analysis
Network flow analysis: detecting threats without full packet capture
Full packet capture provides the richest data for network security analysis but is operationally impractical at scale: the storage requirements and processing overhead of capturing every packet on a 10 Gbps link are substantial. NetFlow and its successors (IPFIX, sFlow) provide a practical alternative by capturing metadata about every network flow — source and destination IP and port, protocol, bytes transferred, packet count, and duration — without capturing payload content. This metadata is sufficient to detect many threat behaviors: unusual communication patterns (a workstation communicating with 200 internal hosts in five minutes indicates scanning), volume anomalies (an endpoint transferring 50 GB to a cloud storage service over two hours indicates potential exfiltration), connection patterns (repeated short-lived connections to the same external IP from multiple internal hosts indicates C2 beaconing), and protocol anomalies (DNS queries that are unusually large or frequent may indicate DNS tunneling).
Network detection and response: behavioral analytics for network traffic
Network detection and response (NDR) platforms apply machine learning and behavioral analytics to network traffic metadata to identify threats that evade signature-based detection. NDR platforms establish behavioral baselines for each device and peer group — what communication patterns are normal for a specific workstation, server type, or department — and generate alerts when observed behavior deviates significantly from the baseline. High-value NDR detection use cases: east-west scanning (a server probing internal hosts on ports it has never accessed before), credential-based lateral movement (authentication attempts to many internal hosts from a single source in a short window), C2 beaconing (regular periodic outbound connections to a single external destination at consistent intervals), and data staging (large internal transfers to a specific endpoint followed by large outbound transfers from that endpoint). NDR provides network visibility that complements EDR's endpoint visibility, covering devices without agents.
DNS security and monitoring
DNS is a foundational protocol that every device uses for every internet communication, making it both a critical monitoring point and an attractive channel for attackers. DNS monitoring for security purposes examines: query volume per device (unusually high query rates may indicate DNS tunneling or malware), query patterns (domains queried that were registered within the past day or week are higher risk than established domains), NX domain responses (failed resolution queries that may indicate C2 domain generation algorithm activity or malware attempting to reach decommissioned C2 infrastructure), and specific domain category blocking (blocking DNS resolution for known malicious domains, newly registered domains without business justification, and domains associated with threat actor infrastructure). DNS security platforms such as Cisco Umbrella, Palo Alto DNS Security, and similar products provide category-based DNS filtering and security monitoring as a layer that operates independently of endpoint security.
Intrusion detection and prevention systems: signature and anomaly approaches
Intrusion detection systems (IDS) and intrusion prevention systems (IPS) inspect network traffic for known attack patterns and anomalous behavior. Signature-based detection matches traffic against a database of known attack signatures — specific byte sequences, protocol anomalies, and behavioral patterns associated with known malware families and attack tools. Signature detection is effective for known threats and generates low false positive rates but requires continuous signature updates and misses novel attacks without matching signatures. Anomaly-based detection establishes a baseline of normal traffic patterns and generates alerts for statistically significant deviations. Anomaly detection can identify novel attacks but produces higher false positive rates that require analyst investigation. Modern IPS platforms combine both approaches, applying signatures for known threats while using anomaly detection as a second layer. Inline IPS placement allows automatic blocking of detected threats; out-of-band IDS placement provides detection without the risk of legitimate traffic being blocked by false positives.
— VPN and Remote Access Security
Securing remote connectivity to organizational resources
VPN architecture and split tunneling considerations
Virtual private networks provide encrypted tunnels for remote access to internal resources. Full tunnel VPN routes all traffic — including internet traffic — through the corporate network, providing complete visibility and control over remote user traffic but increasing latency and bandwidth costs for internet-bound traffic. Split tunnel VPN routes only traffic destined for internal resources through the corporate tunnel, allowing internet traffic to go directly from the user's location. Split tunneling improves user experience and reduces VPN infrastructure load but eliminates visibility into internet traffic from remote endpoints: a compromised remote device can establish malware C2 communications that bypass all corporate security controls. The split tunneling configuration requires careful definition of what traffic is routed internally and what bypasses the tunnel. Inverse split tunneling — route all traffic through the tunnel except explicitly excluded categories — provides a middle ground between full and split tunnel approaches.
Always-on VPN and ZTNA: moving beyond traditional remote access
Always-on VPN enforces VPN connectivity from the moment a device boots, before user login, ensuring that device management traffic and endpoint compliance checks are always routed through corporate infrastructure regardless of the user's location. Zero Trust Network Access (ZTNA) replaces VPN as the remote access model for application access. Instead of giving remote users network-level access to the internal network (where they can reach anything on the network), ZTNA provides application-level access — remote users access specific applications through a brokered connection that verifies their identity, device health, and application access entitlement before establishing a session. The user never has network-level access to the internal network; they can only reach the specific applications they are authorized for. ZTNA eliminates the lateral movement opportunity that VPN provides: a compromised endpoint on a ZTNA connection cannot scan internal networks or reach resources it is not explicitly authorized to access.
VPN appliance security: patching and attack surface management
VPN appliances are among the most actively targeted internet-facing systems in enterprise environments, because they provide direct authenticated access to the internal network and are often inadequately patched. The attack pattern is consistent across incidents: a critical vulnerability is disclosed in a VPN vendor's appliance, organizations delay patching because VPN downtime is operationally disruptive, and threat actors exploit the vulnerability before patching is complete. Major incidents attributable to unpatched VPN vulnerabilities include breaches of critical infrastructure, healthcare organizations, and financial institutions. VPN appliance security requires: prompt patching on a dedicated fast-track schedule rather than the standard 30-day patch cycle, monitoring of vendor security advisories with automated alerting on new CVEs, configuration audit against vendor hardening guides, and consideration of whether the VPN appliance is the appropriate architecture for the organization's remote access needs or whether ZTNA would provide equivalent functionality with reduced attack surface.
Remote access monitoring: detecting misuse and compromise
Remote access authentication events are high-value security monitoring signals because legitimate remote access follows predictable patterns for each user. Anomalies that warrant investigation include: authentication from a country or region where the user has never connected before, authentication at a time significantly outside the user's historical pattern (a user who always connects from 8 AM to 6 PM EST authenticating at 3 AM UTC), simultaneous sessions from geographically distant locations (impossible travel), authentication success after a series of failures (potential credential stuffing success), and VPN sessions that remain active for unusually long periods without activity (potential session hijacking). SIEM correlation rules should alert on impossible travel and new-country authentication automatically. For VPN systems, monitoring should also track which internal resources are accessed during each session — unexpected access to sensitive resources outside the user's normal work pattern indicates potential compromise or insider threat.
— Summary
Key concepts from this module
Network segmentation — zones, micro-segmentation, and zero trust — limits lateral movement
Layered network zones require attackers to cross authenticated boundaries between segments. Micro-segmentation restricts east-west traffic between workloads within the same segment. Zero trust removes implicit network-location trust, requiring explicit authentication and authorization for every access request regardless of origin.
Firewall hygiene requires egress filtering, rule audits, and DMZ exposure minimization
NGFW adds application awareness, identity integration, and TLS inspection. Rule sets accumulate over time and require periodic audit to remove expired access paths. Egress filtering prevents compromised systems from freely establishing C2 and exfiltration. DMZ architecture isolates internet-facing services from the internal network.
Network monitoring uses flow analysis, NDR behavioral analytics, DNS monitoring, and IDS/IPS
NetFlow metadata detects scanning, exfiltration, and beaconing without full packet capture overhead. NDR establishes behavioral baselines and detects deviations. DNS monitoring catches C2 tunneling, DGA activity, and newly registered domain access. IDS/IPS combines signature and anomaly detection.
VPN security requires prompt patching, egress visibility decisions, and ZTNA as a modern alternative
VPN appliances are actively targeted; fast-track patching is required. Split tunneling creates blind spots for remote endpoint traffic. ZTNA provides application-level access without network-level exposure, eliminating lateral movement opportunity. Remote access monitoring should alert on impossible travel and new-country authentication.
