— Credential Attack Techniques
How attackers obtain and exploit passwords at scale
Offline hash cracking: speed advantages of GPU-accelerated attacks
When attackers obtain a database of hashed passwords through a breach, they do not need to submit guesses to the live authentication system. They work offline against the hash file using GPU-accelerated cracking tools such as Hashcat. A single modern gaming GPU can test billions of MD5 hashes per second, or tens of millions of bcrypt hashes per second. The attacker applies dictionary attacks — wordlists of common passwords combined with transformation rules (appending numbers, capitalizing first letters, substituting characters) — then brute force for remaining hashes. Hashing algorithms designed for storage — MD5, SHA-1, unsalted SHA-256 — are trivially cracked. Password-specific algorithms — bcrypt, scrypt, Argon2 — are designed to be computationally expensive, limiting the attacker to thousands of guesses per second. The practical implication: any password that appears in a common wordlist or follows predictable transformation patterns will be cracked if the hash is obtained.
Credential stuffing: automated exploitation of reused passwords
Credential stuffing exploits a fundamental behavioral reality: most users reuse passwords across multiple services. When a breach exposes username-password pairs from one service, attackers test those pairs against every major service where the user might have an account. This is not brute force — it uses the actual compromised credentials. Stuffing attacks run through residential proxy networks to distribute requests across millions of IP addresses, evading rate limiting and IP-based blocking. Success rates for stuffing campaigns against major services range from 0.1% to 2% — low percentages, but against billions of credential pairs from aggregated breach databases, this produces millions of successful account takeovers. Organizations detect stuffing through anomalous authentication patterns: many accounts with low individual failure counts, distributed source IPs, unusual geographic distribution, and login attempts that exactly match times of major breach disclosures.
Kerberoasting: extracting and cracking service account credentials
Kerberoasting targets Windows Active Directory service accounts. Any authenticated domain user can request a Kerberos service ticket for any service registered with a Service Principal Name (SPN). The ticket is encrypted with the service account's password hash. The attacker requests tickets for service accounts — particularly high-privilege ones — exports them, and cracks the password hash offline. Service accounts are particularly vulnerable because they often have non-expiring passwords set years ago before strong password policies were in place, are sometimes configured with weak or guessable passwords, and frequently have excessive privileges. Detection requires monitoring for large volumes of TGS ticket requests from a single account, particularly for service accounts not used by the requesting user's normal workflow.
Pass-the-hash and Kerberos delegation attacks
Pass-the-hash attacks exploit how Windows NTLM authentication works. NTLM uses the password hash directly for authentication — the hash is the credential, not just a representation of it. An attacker who extracts NTLM hashes from memory using Mimikatz can authenticate as that user to any NTLM-based service without knowing the actual password. The hash cannot be used against modern Kerberos-based authentication, but NTLM is still supported across most Windows environments for backward compatibility. Kerberos unconstrained delegation is a configuration that allows a service to impersonate any user to any other service — attackers who compromise a server configured with unconstrained delegation can capture Kerberos tickets for any user who authenticates to that server and use them to impersonate those users. These attacks motivate the defense-in-depth principle: protecting the hash is as important as protecting the password.
— Modern Password Policy
NIST guidance and evidence-based controls
NIST SP 800-63B: what the current guidance actually says
NIST Special Publication 800-63B Digital Identity Guidelines, last substantially updated in 2017 and revised in 2024, represents the most authoritative evidence-based password policy guidance available. Its key positions contradict several widely deployed legacy policies. Mandatory periodic password rotation — the ubiquitous requirement to change passwords every 60 or 90 days — is not recommended unless there is evidence of compromise. Research demonstrates that forced rotation leads users to choose predictable variations (adding a number to the end, capitalizing differently) rather than genuinely different passwords, and creates operational burdens without security benefit. Complexity requirements mandating uppercase, lowercase, numbers, and symbols — another legacy standard — are similarly not recommended as the primary policy because they produce predictable character substitution patterns that do not meaningfully increase resistance to cracking.
What NIST recommends instead: length and breached password screening
NIST 800-63B's positive recommendations focus on two properties that meaningfully increase resistance to attack. Password length is the most important factor in resisting offline cracking: each additional character multiplies the keyspace that must be searched. NIST recommends allowing passwords up to at least 64 characters and not imposing arbitrary truncation. Breached password screening — checking submitted passwords against databases of previously exposed credentials — is the highest-value control. A password of any complexity that has appeared in a breach database is immediately available to attackers via credential stuffing. The Have I Been Pwned (HIBP) API provides a privacy-preserving k-anonymity interface for checking passwords against more than 800 million compromised credentials without transmitting the actual password to the external service. Organizations implementing breached password screening prevent the most common class of password-based attacks at the authentication boundary.
Smart lockout: balancing brute force protection with usability
Account lockout is the traditional defense against online brute force attacks — after a number of failed attempts, the account is locked. Traditional lockout creates a denial-of-service vulnerability: an attacker who knows valid usernames can lock out every account in an organization, disrupting legitimate users. Smart lockout, implemented by major identity platforms including Microsoft Entra ID, applies progressive lockout thresholds that account for contextual signals. Familiar locations and devices experience more lenient thresholds than unfamiliar ones. Lockout affects specific source IP ranges rather than the account globally. Risk-based lockout increases thresholds when attack patterns consistent with credential stuffing (many accounts, distributed IPs, low failure rates per account) are detected, without locking accounts that real users need to access. The combination prevents automated attacks without enabling targeted denial-of-service against individual accounts.
Privileged access management and just-in-time access
Privileged access management (PAM) addresses the credential risk posed by accounts with elevated permissions. PAM systems — CyberArk, BeyondTrust, Delinea — manage privileged credentials by checking them out for specific sessions, rotating them after each use, monitoring the session for unauthorized commands, and recording all activity. Just-in-time access goes further: rather than maintaining standing privileged accounts that exist continuously and could be compromised, JIT access grants elevated permissions only when needed, for a defined duration, approved through a workflow, and revoked automatically when the time expires. This eliminates the attack surface of persistent privileged accounts: an attacker who compromises credentials has a narrow window in which elevated permissions exist to exploit. For organizations subject to NY DFS 500, PAM deployment is a direct regulatory requirement under the privileged access control provisions.
— Authentication Monitoring
Detecting credential attacks through log analysis
Identifying credential stuffing through authentication log patterns
Credential stuffing produces a distinctive authentication log signature that differs from both normal user activity and targeted brute force. Characteristics include: a large number of authentication attempts distributed across many different accounts, each account showing only a small number of failures before either succeeding or moving on, source IPs from residential proxy ranges or distributed across many countries, timing patterns consistent with automated tools (regular intervals, high volume sustained over hours), and correlation with breach disclosure timing — stuffing campaigns often begin within hours of a new breach database appearing on dark web markets. SIEM correlation rules that alert on these patterns — many accounts with low individual failure counts and distributed source IPs over a short window — provide early detection of stuffing campaigns targeting the organization.
Detecting password spraying versus brute force
Password spraying and brute force are distinct attack techniques with different log signatures. Brute force targets a single account with many password attempts in rapid succession — it generates a large number of failures for one account and triggers traditional lockout. Password spraying tests a small number of common passwords — often just one or two — against every account in the organization, staying below lockout thresholds for any individual account. Log analysis for spraying looks for a small number of failures per account distributed across many accounts from the same source IP or IP range within a short window. In Active Directory environments, Event ID 4625 (failed logon) aggregated by source IP and time window reveals spraying patterns that would be invisible when looking at any individual account's log.
OAuth device code phishing: a modern credential capture technique
OAuth device code phishing abuses the OAuth 2.0 device authorization flow — designed for devices like smart TVs that cannot display a browser — to capture authentication tokens without ever seeing the victim's password. The attacker initiates an OAuth device code flow against a target service (Microsoft 365, Google), receives a device code and user code, then sends the victim a phishing email asking them to visit the device activation URL and enter the user code. The victim authenticates with their real credentials and MFA at the legitimate service — everything appears correct because they are on the real site. Their authentication token is then delivered to the attacker's polling request. Detection requires monitoring for OAuth application consent events, particularly for applications the organization has not registered, and device code flow initiations that are not correlated with known device enrollment activities.
Session token security: what happens after successful authentication
Authentication is only the beginning of an access control problem. After a user authenticates successfully, a session token is issued that represents that authentication for subsequent requests. Session token security properties that matter include: appropriate token lifetime (tokens that live for weeks or months create long windows of exposure if stolen), binding to device or network context (tokens that can only be used from the device and network that generated them resist theft), secure transmission (tokens must only traverse encrypted connections), and revocation capability (the ability to invalidate all sessions for a compromised account immediately). Incident response to credential compromise must include immediate session revocation — changing a password does not invalidate existing sessions in all platforms, meaning an attacker with a stolen session token may retain access even after the victim's password is reset.
— Identity Governance
Managing the full credential lifecycle
Joiner-mover-leaver processes and access provisioning
Identity governance manages account and access rights through the full employee lifecycle. Joiner processes provision accounts and access rights when employees are hired — the risk is over-provisioning, granting access beyond what the role requires because it is easier to grant everything than to assess what is needed. Mover processes update access when employees change roles — the risk is accumulation of historical access from previous roles that was never removed. Leaver processes disable and eventually delete accounts when employees depart — the risk is orphaned accounts that remain active after the employee has left, which provide ongoing access to former employees or to attackers who obtain those credentials. Each phase requires defined processes, responsible owners, and technical automation where possible — manual leaver processes are a known source of orphaned account risk.
Access certification: periodic review of user entitlements
Access certification (also called access review or entitlement review) is the periodic process of having account owners or business unit managers verify that users' access rights remain appropriate. Because access accumulates through normal business activity — project access granted for temporary work, expanded access during a system transition, emergency access granted and not revoked — periodic review catches what provisioning processes miss. Certification campaigns should be risk-weighted: privileged access should be certified quarterly or more frequently; standard user access can be certified annually. Automated platforms (SailPoint, Saviynt, Microsoft Entra ID Governance) generate review campaigns, route them to responsible managers, track completion, and revoke access that is not affirmatively recertified within the review window. Access that cannot be mapped to a business justification should be revoked.
Automated deprovisioning and orphaned account detection
Orphaned accounts — accounts that remain active after the user they belong to has left the organization or changed roles — are a persistent security risk. In large organizations with manual provisioning processes, leaver accounts may remain active for weeks or months after departure. Automated deprovisioning integrated with the HR system triggers account disablement on the last working day without requiring an IT ticket. Beyond leaver accounts, additional orphaned account categories include: service accounts for decommissioned applications, test accounts created during development projects, contractor accounts not tied to a contract end date, and shared accounts with no identifiable owner. Automated discovery of accounts not associated with active HR records, combined with aging reports that flag accounts with no recent activity, supports systematic orphaned account cleanup.
Privileged account hygiene and separation of duties
Privileged account hygiene addresses the risk that administrative accounts are used for daily work activities that expose them to compromise. Best practice requires that administrators have two accounts: a standard user account for email, web browsing, and productivity applications, and a separate privileged account used exclusively for administrative tasks. The privileged account should not have an associated mailbox, should not be used to browse the web, and should not be the account used to consume any content that could deliver malware. Separation of duties prevents any single individual from having the access required to both authorize and execute a sensitive operation — a financial control that prevents fraud but equally important as a security control that prevents insider threat and limits blast radius if an administrative account is compromised.
— Summary
Key concepts from this module
Credential attacks: offline cracking, stuffing, Kerberoasting, and pass-the-hash exploit different vulnerabilities
GPU cracking exploits fast hash algorithms. Stuffing exploits password reuse. Kerberoasting exploits SPN ticket encryption with service account password hashes. Pass-the-hash exploits NTLM using extracted hashes as credentials. Each attack targets a different layer of the credential chain.
NIST 800-63B recommends length and breached password screening — not forced rotation or complexity
Forced rotation produces predictable variations without security benefit. Complexity mandates produce character substitution patterns. HIBP k-anonymity API enables breached password screening without exposing the password. Smart lockout balances brute force protection against denial-of-service risk.
Authentication log analysis detects stuffing (many accounts, low failures, distributed IPs) and spraying (few passwords, many accounts)
OAuth device code phishing captures tokens without seeing passwords by abusing legitimate OAuth flows. Session token security requires appropriate lifetime, context binding, and immediate revocation capability on compromise.
Identity governance requires joiner-mover-leaver processes, access certification, automated deprovisioning, and privileged account separation
Access accumulates through normal activity and must be periodically reviewed and recertified. Orphaned accounts from leavers and decommissioned systems are a persistent risk addressed by HR-integrated automation. Separate privileged accounts and separation of duties limit blast radius.
