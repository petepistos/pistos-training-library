— Browser Security and Web Threats
Understanding browser-based attacks and protective controls
Drive-by downloads and exploit kits
Drive-by downloads deliver malware to visitors of compromised or malicious websites without any intentional download action by the user. The attack chain begins when a user visits a website — which may be a legitimate site that has been compromised, or a malicious site reached through a phishing link or malvertising. The page loads JavaScript or HTML that identifies the user's browser version and installed plugins, selects an exploit appropriate for the detected environment, and executes it. The exploit leverages a vulnerability in the browser itself, a browser extension, or a plugin such as PDF readers to execute code. Successful exploitation downloads and executes a malware payload. Exploit kit infrastructure — RIG, Fallout, Magnitude — automates exploit selection and payload delivery. The primary defenses are: keeping browsers and plugins current to eliminate known vulnerabilities, disabling or removing plugins that are not needed, enabling browser sandboxing, and using DNS and web filtering to block known exploit kit infrastructure.
Browser extensions: the overlooked attack surface
Browser extensions are frequently underestimated as a security risk. Extensions run with elevated privileges inside the browser — they can read and modify all page content, access browsing history, intercept network requests, and read form inputs including credentials entered into web forms. Malicious extensions are distributed through official browser extension stores despite review processes, and legitimate popular extensions are acquired and converted to malicious purposes after the original developer sells them. Extension acquisition attacks target developers of popular extensions with phishing campaigns to obtain their store credentials, then publish a malicious update that reaches all existing users automatically. Users rarely review what permissions browser extensions request or monitor them after installation. Browser extension security policies in enterprise environments should: inventory all installed extensions, enforce an allowlist of approved extensions using browser policy, and prevent installation of extensions from sources other than the official store.
Web skimming and formjacking
Web skimming — also known as formjacking or Magecart-style attacks — injects malicious JavaScript into websites that process payment card data or other sensitive form inputs. The injected script runs in the browser alongside the legitimate page content, intercepting form field values (card numbers, expiration dates, CVV codes, billing addresses) as users type them and transmitting them to attacker-controlled servers. The legitimate payment processing continues normally — the user's transaction completes, but their card data has been copied. The injected code may be placed directly in the target site, in a third-party library the site loads, or in a fourth-party service that the third-party library loads — supply chain injection allows a single compromise to affect hundreds of sites simultaneously. Detection is difficult for users and challenging even for the site operators because the skimming code is designed to be invisible during normal operation.
Content security policy and browser security headers
Web security headers allow web application developers to instruct browsers to enforce security restrictions that limit the impact of common vulnerabilities. Content Security Policy (CSP) specifies which sources of scripts, styles, images, and other resources the browser is permitted to load for a given page. A strict CSP that allows scripts only from the site's own origin prevents injected third-party scripts — including web skimmers — from executing, because the browser refuses to load scripts from sources not on the allowlist. HTTP Strict Transport Security (HSTS) instructs browsers to always use HTTPS for a domain and to refuse connections that cannot be secured, preventing protocol downgrade attacks. X-Frame-Options prevents the page from being embedded in an iframe, defeating clickjacking attacks. These headers require explicit configuration by web developers and are absent from many sites, leaving users relying on browser-level protections alone.
— Cloud Security and SaaS Risk
Protecting data and access in cloud and SaaS environments
Shared responsibility model: what the cloud provider secures and what you must secure
The shared responsibility model defines the division of security obligations between the cloud provider and the customer. The cloud provider is responsible for the security of the cloud infrastructure: physical data centers, network hardware, hypervisors, and the underlying computing platforms. The customer is responsible for security in the cloud: operating system configuration and patching (for IaaS), application security, identity and access management, data classification and protection, and network configuration within the cloud environment. The boundary shifts depending on the service model: in SaaS, the provider manages nearly everything, and the customer is responsible primarily for user access configuration and data governance. Misunderstanding or ignoring this boundary is a leading cause of cloud security incidents — organizations that assume the provider handles all security are not configuring the controls they are responsible for.
OAuth and third-party application access risks
OAuth 2.0 authorization allows users to grant third-party applications access to their accounts without sharing passwords. When a user clicks 'Sign in with Google' or 'Connect to Microsoft 365' in a third-party application, they are granting that application defined permissions to their account. The OAuth consent screen lists what permissions are requested — read email, access files, send email on the user's behalf — but users rarely scrutinize these permissions before granting consent. Organizations that do not manage OAuth application access allow an unbounded proliferation of third-party applications with access to organizational data. Risky OAuth patterns include: applications requesting more permissions than their stated function requires, applications from unrecognized developers with few installs, and applications that request continuous access rather than read-only access. In Microsoft 365, tenant-wide OAuth consent policies should require admin approval for applications requesting access to organizational data.
Shadow IT: unauthorized cloud services and data exfiltration
Shadow IT refers to cloud services, applications, and tools used by employees without organizational knowledge, approval, or security review. When employees find that approved tools do not meet their needs, they adopt alternatives — transferring files through personal Dropbox accounts, using personal Gmail for business communications, connecting unapproved AI tools to organizational data. Each shadow IT instance creates: data governance risk (organizational data in an uncontrolled third-party environment), security risk (credentials for these services are managed outside organizational security controls), compliance risk (regulated data in unapproved environments), and discovery risk (the organization cannot account for where its data resides). CASB (Cloud Access Security Broker) platforms provide visibility into cloud service usage through analysis of network traffic metadata, allowing discovery of shadow IT usage and enforcement of policies that block unapproved services or require routing through controlled channels.
Cloud identity and access management hygiene
Cloud environments enable rapid creation of users, service accounts, and access policies, which leads to identity sprawl if not actively managed. Common cloud IAM hygiene failures include: excessive permissions granted because the least-privilege principle is difficult to implement precisely, access keys for service accounts that have not been rotated and may have been exposed, root account usage in cloud environments where the root account should be locked down and never used for routine operations, and access policies that use wildcard permissions rather than specific actions and resources. Cloud security posture management (CSPM) tools continuously assess the cloud environment's configuration against security best practices, generating findings for misconfigurations including overly permissive IAM policies, publicly accessible storage buckets, and unencrypted sensitive data stores. Regular CSPM findings review, combined with automated remediation for high-confidence findings, maintains cloud IAM hygiene as the environment scales.
— Secure Web Gateway and Content Filtering
Controlling and monitoring internet access
Web proxy and URL filtering: categories and enforcement
Secure web gateways (SWG) intercept all web traffic from internal users and apply policy before allowing or blocking access. URL category filtering classifies websites into categories — productivity tools, social media, gambling, adult content, malware distribution, newly registered domains — and applies allow, block, or warn policies by category. Organizations configure category policies to match their acceptable use policies and security requirements: blocking malware and phishing categories entirely, warning for social media categories, and logging all access for forensic purposes. SWG also applies SSL inspection to HTTPS traffic, decrypting and inspecting content before re-encrypting and forwarding, allowing content inspection of encrypted traffic. The combination of category filtering, SSL inspection, and threat intelligence integration creates a web security layer that reduces the attack surface available to users browsing the internet and provides visibility into web activity that would otherwise be invisible in encrypted traffic.
DNS filtering: blocking at resolution rather than connection
DNS-based web filtering blocks access to malicious or policy-violating domains at the DNS resolution layer — before a connection is ever established. When a user's device attempts to resolve a domain that is blocked by policy, the DNS filter returns a block page address instead of the real IP, preventing the connection from forming. DNS filtering provides security value that differs from URL filtering: it is protocol-agnostic (it blocks malware C2 domains used by any protocol, not just web browsers), it is faster to implement (no SSL inspection certificate distribution required), and it provides coverage for devices that cannot have proxy certificates installed. Cloud-delivered DNS security platforms like Cisco Umbrella, Cloudflare Gateway, and similar services provide DNS filtering with threat intelligence integration, reporting, and policy management. Enforcing DNS filtering requires ensuring all devices use organizational DNS resolvers and cannot bypass filtering by configuring alternative DNS servers.
Threat intelligence integration in internet security controls
Internet security controls are most effective when enriched with current threat intelligence — continuously updated information about known malicious infrastructure, active phishing campaigns, malware distribution sites, and attacker-controlled domains. Threat intelligence feeds integrate with web gateways, DNS filters, firewalls, and SIEM platforms to provide real-time blocking of known threats and detection of communication with identified attacker infrastructure. Free threat intelligence sources include: CISA advisories and known exploited vulnerabilities catalog, FBI flash alerts, abuse.ch (which tracks malware and botnet infrastructure), Spamhaus (which tracks spam, phishing, and malware domains), and VirusTotal (which aggregates antivirus verdicts for files and URLs). Commercial threat intelligence platforms aggregate and normalize feeds from multiple sources with higher update frequency and lower false positive rates. The combination of free and commercial intelligence sources provides breadth and depth.
Personal device and BYOD internet security risks
Bring-your-own-device policies create internet security challenges because personal devices operate in both personal and professional contexts that have different security requirements and different levels of organizational control. Personal devices may have: unpatched operating systems and applications, browser extensions that capture credentials entered into corporate applications, personal use that exposes the device to threats that would then have access to corporate data and applications, and no endpoint protection software meeting organizational standards. Mobile device management (MDM) and mobile application management (MAM) provide different levels of organizational control over personal devices — from full device management (high control, high privacy impact) to application-level containerization (controls only the work applications, does not monitor personal use). BYOD policy should define what data classifications are permitted on personal devices, what controls are required, and what happens to organizational data if the device is lost or stolen.
— Incident Response for Internet-Based Threats
Responding to web-based attacks and data exposure
Web application attack response: identifying and containing exploitation
Web applications exposed to the internet are a primary attack surface and a common initial access vector. When web application exploitation is suspected — through WAF alerts, anomalous application logs, or security scanner findings — the incident response begins with determining: what vulnerability was exploited (specific CVE, application logic flaw, injection vulnerability), whether exploitation resulted in unauthorized data access or code execution, and whether a persistent backdoor (webshell) was installed. Webshell detection requires examining the web server's file system for recently modified or created files in web-accessible directories, particularly scripts with names that blend with legitimate application files. Web application firewall (WAF) logs provide exploitation attempt patterns. Application logs show authentication anomalies, unusual query patterns, and error conditions consistent with exploitation. Containment may require taking the application offline, blocking specific attack patterns in the WAF, or deploying temporary mitigations while a permanent patch is developed.
Data exposure incidents from cloud misconfiguration
Cloud storage misconfiguration — particularly publicly accessible storage buckets containing sensitive data — is among the most common data exposure incident types. Response to a discovered cloud storage exposure requires: immediately correcting the access control configuration to remove public access, determining how long the exposure existed by examining bucket creation and access control change logs, assessing what data was exposed by inventorying bucket contents, determining whether the data was accessed by analyzing access logs for GET requests to sensitive objects during the exposure window, and assessing regulatory notification obligations based on the nature of the exposed data and the applicable regulations. For insurance agencies subject to NY DFS 500, exposure of NPI triggers notification requirements with specific timelines. Cloud storage access logs provide granular per-object access history if logging was enabled — organizations should verify that access logging is enabled on all storage buckets containing sensitive data before an incident requires them.
Malvertising and watering hole attack response
Malvertising (malicious advertising) and watering hole attacks deliver malware through websites that users visit in the course of normal activity — legitimate news sites, industry publications, and professional forums that serve malicious ads or have been compromised to deliver exploit code. Response to these incidents requires: identifying which users visited the affected site during the window when malicious content was served, assessing whether any of those visits resulted in successful exploitation based on endpoint detection telemetry from the same time window, triaging affected endpoints for malware indicators, and communicating with affected users about what to look for and how to report symptoms. DNS query logs and web proxy logs provide the list of users who visited the affected site. EDR telemetry shows exploitation attempts and subsequent malicious activity. Watering hole incidents may require notifying the compromised site operator about their compromise.
Credential exposure and dark web monitoring
Credentials exposed through breaches, phishing campaigns, and info-stealer malware appear on dark web markets and paste sites where they are traded, sold, and used for credential stuffing attacks. Dark web monitoring services watch these sources for credentials associated with the organization's domains, alerting when email addresses and associated passwords appear in breach data or are offered for sale. When exposed credentials are identified: force password resets for all exposed accounts, assess whether the exposed accounts were used in credential stuffing or other attacks between the exposure and the notification, check for unauthorized access from the same time window, and revoke all active sessions for affected accounts. Organizations should configure their identity platform to check new and changed passwords against breach databases (HIBP integration) to prevent reuse of credentials that are already in attacker databases. Dark web monitoring is a complementary detective control that catches exposure that is not prevented by preventive controls.
— Summary
Key concepts from this module
Browser threats include drive-by downloads, malicious extensions, web skimming, and CSP-preventable injection
Exploit kits target unpatched browser plugins and extensions. Malicious browser extensions read all page content including credentials. Web skimmers inject JavaScript that copies form inputs to attacker servers. Content Security Policy prevents unauthorized script execution. Keeping browsers and extensions current is the foundational defense.
Cloud security is shared responsibility; OAuth, shadow IT, and IAM sprawl are the primary risk areas
Customers are responsible for identity, access, and data configuration regardless of service model. OAuth grants third-party apps access to organizational data — require admin consent for new apps. CASB discovers shadow IT usage. CSPM continuously assesses cloud IAM and configuration against best practices.
SWG provides URL filtering and SSL inspection; DNS filtering blocks at resolution and covers all protocols
Threat intelligence integration enriches both controls with current attacker infrastructure data. BYOD requires MDM or MAM to enforce minimum security requirements for devices accessing organizational resources.
Internet incident response covers web app exploitation, cloud misconfiguration, malvertising, and credential exposure
Web app response: identify vulnerability, find webshells, contain via WAF and patch. Cloud misconfiguration: correct access controls, inventory exposed data, check access logs, assess regulatory notification. Credential exposure: force resets, session revocation, dark web monitoring for ongoing detection.
