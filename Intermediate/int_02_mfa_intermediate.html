— How MFA Is Defeated
Attack techniques that bypass multifactor authentication
Real-time phishing proxies relay credentials and OTP codes simultaneously
Adversary-in-the-middle phishing frameworks — Evilginx2, Modlishka, Muraena — operate as transparent proxies between the victim and the legitimate service. The victim sees a pixel-perfect clone of the real login page served through the attacker's domain. When the victim enters their username, password, and one-time code, the proxy relays those credentials to the real site in real time, completes authentication, and captures the resulting session cookie. The attacker then uses that session cookie to access the account — the OTP has already been consumed and is worthless for replay, but the authenticated session persists. This attack defeats SMS codes, authenticator app TOTP, and push notifications equally. Only FIDO2/WebAuthn hardware keys are immune because the authentication is cryptographically bound to the legitimate domain.
SIM swapping transfers a victim's phone number to an attacker-controlled SIM
SIM swapping exploits the mobile carrier's account recovery and number porting processes. An attacker calls the carrier posing as the account holder, typically armed with personal information gathered from data breaches or social media. They convince the carrier representative to transfer the target's phone number to a SIM card the attacker controls. Once the swap completes — which can take minutes — all SMS messages and calls to that number route to the attacker's device. Any account using SMS-based MFA can then be compromised: the attacker triggers a password reset or MFA challenge, receives the code on the now-controlled number, and completes authentication. High-profile SIM swap attacks have targeted cryptocurrency holders, journalists, and executives with access to sensitive accounts.
MFA fatigue exploits push notification approval to bypass authentication
MFA fatigue, also called prompt bombing, exploits the push notification mechanism in authenticator apps like Microsoft Authenticator and Duo. An attacker with valid credentials repeatedly triggers login attempts, generating a continuous stream of push notifications on the victim's device. The attack relies on the victim eventually approving a prompt — through confusion about whether they initiated a login, irritation at the repeated interruptions, or the mistaken belief that approving will stop the notifications. Several high-profile breaches have been attributed to this technique, including incidents at major technology companies. Number matching — requiring the user to enter a number displayed on the login screen into the push prompt — is the primary technical countermeasure, because it prevents accidental or careless approvals.
SS7 protocol vulnerabilities allow interception of SMS at the network level
The Signaling System 7 protocol is the decades-old telecommunications signaling framework used to route calls and messages between carriers worldwide. SS7 contains fundamental design flaws — it was designed for a closed network of trusted carriers and has no authentication requirements between nodes. Attackers with access to SS7 infrastructure (obtainable through rogue carriers or criminal services) can intercept SMS messages and redirect calls intended for any phone number without the carrier's knowledge and without any interaction from the victim. This capability is used by intelligence agencies for surveillance and by criminals for targeted account takeovers. Unlike SIM swapping, SS7 attacks leave no trace on the victim's device and require no interaction with the carrier. SMS MFA is fundamentally vulnerable to this attack class.
— FIDO2 and WebAuthn in Depth
How phishing-resistant authentication works at the protocol level
FIDO2 uses asymmetric cryptography with domain-bound key pairs
When a FIDO2 credential is registered with a service, the authenticator — a hardware key or platform authenticator — generates an asymmetric key pair: a private key stored securely in the authenticator and a public key registered with the service. The critical security property is that the key pair is generated per-relying-party: the private key for bank.com is mathematically distinct from the private key for email.com, even on the same authenticator. During authentication, the service sends a cryptographic challenge. The authenticator signs that challenge with the private key, and the service verifies the signature using the registered public key. The private key never leaves the authenticator hardware, and no shared secret is ever transmitted that could be phished or stolen from the server.
Origin binding makes FIDO2 immune to phishing and relay attacks
The relying party identifier — the domain of the service being authenticated to — is embedded in the cryptographic challenge sent during authentication. The authenticator will only sign a challenge if the domain in the challenge matches the domain the credential was registered for. If an attacker operates a phishing proxy at evil-bank.com that relays all traffic to bank.com, the challenge the proxy forwards will contain the legitimate bank.com origin. But the request the user's browser sends to the authenticator will specify evil-bank.com as the origin — and the authenticator will refuse to sign because evil-bank.com does not match the registered relying party. This origin binding is enforced at the cryptographic layer and cannot be worked around by the phishing proxy, regardless of how convincing the fake site appears.
Platform authenticators extend FIDO2 to built-in device biometrics
FIDO2 is not limited to external hardware security keys. Platform authenticators use secure hardware built into modern devices — the Trusted Platform Module in Windows PCs, the Secure Enclave in Apple devices, the Trusted Execution Environment in Android devices — to generate and store FIDO2 credentials. Authentication is completed with the device's biometric sensor (fingerprint or face recognition) or PIN. Windows Hello, Apple Face ID/Touch ID for web authentication, and Android's biometric authentication all implement platform authenticators that provide FIDO2/WebAuthn security. Platform authenticators provide the same origin-binding and domain-bound key pair security as external hardware keys but use the biometric bound to the specific device as the user verification method.
Passkeys are FIDO2 credentials designed for consumer-friendly deployment
Passkeys are FIDO2 credentials that address usability friction by enabling synchronization across a user's devices through the platform vendor's cloud infrastructure — iCloud Keychain for Apple devices, Google Password Manager for Android and Chrome, and Windows for Microsoft accounts. A passkey created on one iPhone is available on all of the user's Apple devices, eliminating the 'what if I lose my hardware key' concern. Passkeys provide the same phishing-resistant, domain-bound authentication as hardware FIDO2 keys. They are increasingly being deployed by major consumer services — Google, Apple, Microsoft, GitHub — as a replacement for password-plus-MFA authentication. For enterprise deployment, passkeys introduce questions about device management, credential portability across platforms, and integration with existing identity infrastructure.
— MFA Deployment and Policy
Organizational considerations for effective MFA implementation
Conditional access policies apply MFA requirements based on risk signals
Modern identity platforms — Microsoft Entra ID, Okta, Ping Identity — support conditional access policies that evaluate risk signals at the time of authentication and apply MFA requirements accordingly. Risk signals include: the device's compliance status (managed and patched versus unmanaged), the user's location (on-premises versus remote, familiar country versus unfamiliar country), the sensitivity of the resource being accessed (low-sensitivity application versus administrator console), and behavioral signals (typical login time and pattern versus anomalous). A compliant managed device on the corporate network accessing a low-sensitivity application may not require MFA step-up. The same user accessing an administrator console from an unfamiliar country at 3 AM will require strong MFA regardless of prior authentication. This risk-based approach balances security with usability.
Privileged access workstations and step-up authentication for administrative tasks
Privileged access — the ability to modify system configurations, access sensitive databases, or manage user accounts — represents the highest-risk authentication scenario and warrants the strongest controls. Privileged access workstations (PAWs) are dedicated, hardened devices used exclusively for administrative tasks. They are not used for email, web browsing, or any other activity that could introduce malware. Access to privileged resources from PAWs requires strong MFA — hardware FIDO2 keys are the appropriate standard. Step-up authentication requires re-authentication with a second factor immediately before a privileged action, even if the user is already authenticated to the system. This limits the window during which a compromised session can be used to perform high-impact operations.
MFA exceptions and break-glass accounts require careful controls
Operational reality means that MFA exceptions are sometimes requested: a service account that cannot interactively authenticate, a system that predates MFA support, an emergency break-glass account for use when the primary administrator is unavailable. Each exception represents a security gap that must be compensated. Service accounts should use certificate-based authentication rather than password-plus-MFA where possible. Legacy systems that cannot support MFA should be network-isolated and subject to enhanced monitoring. Break-glass accounts should be stored in a physical safe, have their passwords rotated after every use, generate alerts on any login, and be audited regularly. Every MFA exception should be documented, justified, time-limited where possible, and reviewed periodically.
MFA enrollment security: protecting the registration process
The security of MFA is only as strong as the security of the enrollment process. If an attacker can enroll their own authenticator device on a victim's account, they have effectively added a backdoor that bypasses MFA. Attackers compromise accounts during the MFA enrollment window — before the user has completed setup — or use account takeover to add a second authenticator device. Enrollment security requires: strong identity verification before enrollment is permitted, requiring re-authentication at the time of enrolling a new factor, monitoring for newly enrolled authenticators as a security event, limiting enrollment to managed organizational processes rather than self-service, and immediate notification to the account owner when a new factor is registered. Enrollment events should appear in security logs and generate alerts for review.
— Recognizing and Responding to MFA Attacks
Operational awareness for users and security teams
Identifying a real-time phishing proxy attack in progress
Several observable signals distinguish a real-time phishing proxy attack from legitimate authentication. The URL in the browser address bar contains the phishing domain rather than the legitimate service — the single most reliable indicator. Password managers that store credentials by domain will not offer to autofill, because they do not recognize the domain. The certificate presented may be for a domain similar to the legitimate service but not identical. Some phishing proxy frameworks introduce subtle visual artifacts or timing differences in page load behavior. The most reliable defense remains the single habit of checking the exact domain in the address bar before entering credentials — automated defenses such as password managers and FIDO2 authenticators enforce this check at the technical layer, making it independent of user observation.
What to do when you receive unexpected MFA prompts
An MFA push notification, SMS code request, or TOTP challenge that you did not initiate is a clear signal that someone with your password is attempting to log in. The correct immediate actions are: deny the prompt (never approve), document the time and nature of the prompt, report immediately to the security team, change the account password from a clean device (not the one that may be affected), and review recent account activity for unauthorized access that may have occurred before the MFA prompt appeared. If prompts are repeated in quick succession, this is an active MFA fatigue attack — deny each one explicitly using the available decline option rather than dismissing the notification, as explicit denial sends a signal to some systems that triggers additional security review.
Detecting SIM swap and SS7 attacks on SMS-based MFA
SIM swap attacks produce specific observable signals. The victim's phone suddenly loses service — calls fail, messages do not deliver — while the attacker's device receives them. Some carriers send a text message to the original SIM immediately before the swap, though this protection is inconsistently implemented. Account activity notifications from services using SMS MFA may appear for activity the victim did not initiate. If your phone unexpectedly loses service, contact your carrier immediately using an alternate phone to verify whether a number transfer has occurred. Organizations that depend on SMS MFA should monitor for authentication events that occur after extended periods of no activity on a specific phone number, which may indicate the number has been transferred. Migration to app-based or hardware key MFA eliminates the SIM swap attack surface entirely.
Responding to MFA bypass incidents at the organizational level
When an MFA bypass is suspected or confirmed — a session cookie stolen via phishing proxy, an account compromised through SIM swap, a push notification approved during a fatigue attack — the organizational response requires several parallel actions. Revoke all active sessions for the compromised account immediately. Force re-enrollment of all MFA factors to ensure the attacker has not registered their own device. Audit account activity since the compromise window to identify data accessed or actions taken. Assess whether the compromised account had access that would enable lateral movement to other accounts or systems. Review whether the attack technique affects other users in the organization — a phishing proxy campaign is likely targeting multiple users, and a warning to the broader user population may be warranted. Document the incident for post-incident review and regulatory assessment.
— Summary
Key concepts from this module
MFA bypass attacks — proxy phishing, SIM swap, fatigue, SS7 — each defeat specific MFA implementations
Real-time phishing proxies defeat OTP-based MFA. SIM swap and SS7 attacks defeat SMS MFA. Fatigue attacks exploit push notifications. Only FIDO2/WebAuthn with origin binding is immune to proxy phishing. Understanding which attacks defeat which MFA types determines the appropriate deployment for each risk level.
FIDO2/WebAuthn uses domain-bound asymmetric key pairs — private keys never leave the authenticator
The origin binding in FIDO2 makes phishing proxy attacks cryptographically impossible: the authenticator refuses to sign a challenge for a domain that does not match the registered relying party. Platform authenticators extend this security to built-in device biometrics. Passkeys bring FIDO2 to consumer deployment with cross-device synchronization.
Deployment requires conditional access policies, privileged access controls, and exception management
Risk-based conditional access balances security with usability. Privileged accounts warrant hardware FIDO2 keys and step-up authentication. Every MFA exception must be documented, compensated, and reviewed. Enrollment security is as important as authentication security.
Unexpected MFA prompts are active attacks — deny, document, report, rotate credentials
Never approve an uninitiated MFA prompt. Repeated prompts are a fatigue attack in progress. SIM swap produces sudden loss of mobile service. Organizational response to MFA bypass requires session revocation, factor re-enrollment, activity audit, and lateral movement assessment.
